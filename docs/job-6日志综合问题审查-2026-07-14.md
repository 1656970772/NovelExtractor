## 一、结论摘要

本次 Token 越跑越多，不是因为章节窗口越来越大，也不是因为上一窗口的聊天历史泄漏到下一窗口。

真正的主链路是：

1. 每个新窗口都会重新创建消息列表，首轮输入基本稳定在约 3.4 万～4.3 万 Token。
2. 同一窗口内部的工具循环没有总轮次、累计 Token 或工具返回字符预算。
3. `grep`、`read_file` 和编辑错误会把很长的完整行继续塞回上下文；每轮又完整携带前面所有 assistant/tool 消息。
4. 报告越大、长行越多，单次工具结果越容易放大，后续每一轮请求就越贵。
5. 暂停、失败和重新运行还存在竞态、无限重试与 Token 记账重置，使实际消耗进一步扩大且界面严重低估。

因此，“暂停后首批窗口又很便宜”是符合当前实现的：暂停/重跑后换到新窗口，新窗口首轮上下文重新变小；但只要该窗口继续多轮搜索、读取、编辑，请求大小又会快速累积。

### P0-2：Token 统计重置，失败/进行中窗口消耗漏记

**证据**

- 全部成功模型响应日志累计 `1,169,458,575` Token，当前 `project-runtime.json` 和界面却显示 `Token 0`。
- 上一个紧邻 attempt 的 19 个 batch 日志有 151 次模型返回、累计 `31,837,060` Token；重新运行后当前状态变成 1292 skipped、0 executed、Token 0。
- `windowRunService.ts:3910-3919` 每次 `startJob` 把 usage 初始化为零。
- `windowRunService.ts:4036-4045` 只有整个窗口成功完成后才将 usage 加到状态。
- 窗口失败时，`:3289-3295` 只把已耗 usage 写入日志然后抛出，没有进入任务累计。

**影响**

- 暂停后重新运行、自动续跑或应用重启会丢失任务历史消耗。
- 超长窗口完成之前，界面看不到正在发生的巨大消耗。
- 失败、取消、崩溃和并行 fatal 时的已成功请求永久漏记。

**建议与验收**

- 持久化 `lifetimeUsage`、`attemptUsage` 和 `inFlightUsage`，不要混成一个会清零的字段。
- 每次模型响应后立即、原子地记账，不等待窗口完成。
- 验收：暂停、继续、失败、重启、取消前后的 lifetime Token 单调不减，并可与模型响应日志逐条对账。

### P0-3：删除运行中任务可能留下孤儿 worker

**状态**：代码确定存在；本次日志没有实际删除操作，不能声称已经造成数据损坏。

**证据**

- `windowRunService.ts:4001-4013` 仅在没有活跃窗口或所有窗口都处于 retry wait 时等待 `settled`；正常模型/工具调用仍活跃时直接返回成功。

- 验收：删除返回后，相关日志、coverage 和正式报告的修改时间不能再变化。

### P1-3：暂停竞态与控制不及时

**本次暂停竞态时间线**

1. `20260713-220127.txt` 在 22:01:30 记录 14 个批次同时进入 Token Plan 等待。
2. `project-runtime.json` 在 `2026-07-14T01:16:21.381Z` 变成 paused。
3. 暂停生效同一秒，13 个批次又各发起一次请求并再次失败；只有首个 waiter 真正停住。

**代码根因**

- `windowRunService.ts:3836-3865` 中第一个 waiter 把 `pauseRequested` 清成 false 并进入 paused wait。
- 后续 waiter 只检查 `pauseRequested`，看不到 `state.status === "paused"`，于是直接返回并继续请求。
- 正常工具循环没有 pause/cancel/fatal 检查；暂停或取消不能中断正在进行的 fetch 和后续工具轮次。

**建议与验收**

- 使用任务级共享 pause barrier，不允许某个 waiter 独占并清除共享请求。
- 每轮模型请求、每次工具调用前后检查控制状态；请求层支持 AbortSignal。
- 验收：点击暂停后不再产生下一轮模型请求；多个 retry waiter 全部同时停在 barrier。

### P1-4：确定性错误也会无限重试和自动续跑

**证据**

- 单个最大完整日志约 98.4 MB，没有单文件轮转、目录总量或保留 attempt 数限制。

**建议与验收**

- 默认记录消息数、各角色字符数、hash、usage 和工具摘要；完整 payload 只在显式 debug 模式开启。
- Prompt 和 ProviderBody 只保留一份规范化请求，另一个只记协议差异。
- 配置化单文件轮转、目录预算、保留 attempt 数和压缩策略。
- 验收：相同 50 轮样本的默认日志体积下降至少一个数量级，且仍能还原轮次、工具、usage 和失败原因。

### P2-1：未启用工具造成 170 个浪费轮次

- 日志收到 `bash` 169 次、`exec` 1 次；155 个窗口受影响。
- 默认工具列表没有这两个工具，运行时只返回可恢复错误并继续下一轮。
- 建议在 prompt 中明确无 shell 工具，供应商支持时启用严格工具名约束；同一窗口再次调用未知工具时应快速熔断。

### P2-2：绝对路径回显导致后续范围拒绝

- 实际发生 17 次读取范围拒绝：`read_file` 7 次、`grep` 10 次。
- 部分错误信息把项目内报告的本机绝对路径回显给模型，模型随后原样复用；工具安全层只接受项目相对路径/报告别名，因此拒绝。
- 安全边界没有发现逃逸，这是交互规范化问题，不应通过放宽外部路径访问解决。
- 建议模型可见错误只显示 `[报告]文件名.md`；确认 `realpath` 仍在 reports 根目录后，可将合法绝对路径规范化为别名。

### P2-3：日志分类与可观测性不足

- Token Plan 错误文本是“额度已用尽”，`isRateLimitError` 没有识别该短语，导致 943 条额度等待被记为“执行失败”。
- 顶层日志中没有暂停、继续、取消、删除和统一终态事件，只能靠文件时间与 state 更新时间反推。
- 完整日志使用 UTC 时间却去掉 `Z`/时区；例如日志 `01:16:21` 对应本地 `09:16:21`。
- 646 个 batch 日志的任务头显示“未知书籍、0 个模板、未知模型”；完整日志 App Version 也均为 unknown，只有 commit 可用。
- 简要日志漏记 17 次普通读取范围拒绝原因，部分有错误的窗口显示“多轮原因：无”。

建议补齐：控制事件、首个 fatal、drain、终态、failure category/fingerprint、active/retry waiter 数、累计 usage、时区、App Version、模型、书籍和模板元数据。

### P2-4：缓存命中始终为 0，原因待确认

- 全日志 12,906 个 cache usage 块中，非零 `cacheHitTokens` 为 0。
- usage 字段映射代码看起来正常，但当前 OpenAI Responses 适配器没有 response chaining 或显式缓存控制。
- 这可能是供应商不返回缓存明细，也可能是请求前缀/协议无法命中；仅凭日志不能认定为程序 bug。
- 应先确认 MiniMax-M3 当前接口的缓存能力和字段协议，再决定是否实现稳定前缀或 chaining。缓存不能替代轮次与字符硬预算。

## 七、建议修复顺序

### 第一阶段：先止住不可控消耗和错误记账

1. 每次模型响应立即持久化 lifetime Token，修复重启/失败漏记。
2. 增加每窗口轮次、累计 Token、工具结果字符硬预算。
3. 修复大文件分页读取矛盾。
4. 对 HTTP/工具/窗口/job 四层重试做统一错误分类和次数预算。

### 第二阶段：修复控制与状态一致性

1. 修复 pause barrier 竞态，并把 AbortSignal 下沉到请求与工具边界。
2. 删除任务必须等 worker drain；修复孤儿 worker 风险。
3. 串行化并原子保存 `project-runtime.json`，聚合跳过进度。
4. 修复 `written` 被 `no_update` 降级覆盖。

### 第三阶段：降低正常执行成本

1. 恢复有界的结构化报告读取/更新机制。
2. 为 batch 增加配置化并发 semaphore 和 provider 级额度 gate。
3. 降低锚点编辑、未知工具和重复只读调用造成的轮次。

### 第四阶段：日志与观测

1. 默认停止重复保存完整 Prompt/ProviderBody，增加轮转和保留策略。
2. 补齐控制、终态、失败分类、时区、构建和任务元数据。
3. 与供应商确认缓存协议后，再决定是否优化缓存。

## 八、建议的完成验收基线

- 固定 15 章窗口连续执行时，任何窗口都不会突破配置化轮次/Token/字符预算。
- 暂停后不再出现新的模型请求；取消和删除能中断进行中的请求，删除返回后不再写报告或 coverage。
- 任务 lifetime Token 在暂停、继续、失败、自动续跑和重启后单调不减，与响应日志可对账。
- 大于 1 MiB 的报告能安全小分页读取；单次 grep/read/edit 错误返回均受字符预算限制。
- 确定性 HTTP 400 和稳定工具错误不会无限重试或触发无上限 job 自动续跑。
- `written` 结果不会被 `no_update` 降级。
- `project-runtime.json` 在高并发和进程中断下保持可解析、单调和可恢复。
- 默认详细日志体积至少下降一个数量级，同时保留轮次、工具、usage、控制和失败诊断能力。
