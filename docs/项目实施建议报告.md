## 一、核心问题分析

### 1.1 技术栈不匹配 ⚠️ 严重

**问题描述**:
- 实施计划明确要求使用 **Python 3.11+** + 标准库 + PyYAML + pytest
- 当前 `E:\AI_Projects\NovelExtractor` 是 **DeepSeek-Reasonix 的完整克隆**（Go 1.x 项目）
- 目录包含 `.git`、`go.mod`、`go.sum`、`Makefile`、大量 Go 源码

**影响**:
- 无法直接在 Reasonix 仓库中实施 Python 计划
- Python 包结构（`src/novel_extractor/`）与 Go 项目结构冲突
- `pyproject.toml` 与 `go.mod` 共存会导致开发工具混淆

**根本原因**:
用户期望"参考 Reasonix 的工具调用代码"，但实施计划设计的是完全独立的 Python 项目，不是修改或扩展 Reasonix。

---

### 1.2 "参考 Reasonix 代码"的歧义 ⚠️ 中等

**用户原话**:
> "我希望整个执行流里的工具调用都是直接参考人家的已经有的代码，不要自己造轮子，可以适配项目，但是底层实现直接照搬就行。"

**问题**:
1. **Reasonix 是 Go 实现**，Python 项目无法"直接照搬"代码
2. **工具调用机制差异**:
   - Reasonix: 使用 MCP (Model Context Protocol)，JSON-RPC 子进程通信，Go 编译的内置工具
   - MVP 计划: 简单的 OpenAI-compatible HTTP 客户端，不暴露文件工具给模型
3. **架构差异**:
   - Reasonix: Agent harness，模型驱动的工具调用循环
   - MVP: 批处理控制器，模型只负责语义判断和生成 Markdown

**可能意图**:
用户可能是想参考 Reasonix 的 **架构思路**（配置驱动、进度管理、前台执行），而不是复制代码。

**问题**:

2. MVP 计划是 **自动化批处理**，无人工干预的窗口循环

**价值**:

## 二、实施计划质量评估

### 2.1 优点 ✅

1. **架构设计清晰**:
   - Pipeline 模式、Repository 模式（Ledger）、Catalog 模式
   - 关注点分离：批处理控制器 vs 语义判断
   - 配置驱动，避免硬编码

2. **技术选型合理**:
   - Python 标准库优先，依赖少（PyYAML、openai）
   - SQLite 进度管理，支持中断续跑
   - Markdown 业务输出，避免 JSON 解析脆弱性

3. **MVP 范围控制严格**:
   - 只实现 5 个模板验证
   - 不做 UI、并发、向量化、后台服务
   - 前台执行，绑定控制台生命周期

4. **TDD 驱动**:
   - 每个任务先写测试，再实现
   - 测试覆盖配置、章节解析、模板路由、进度管理、文档索引、写入校验

5. **中文环境适配**:
   - 明确 UTF-8 编码
   - PowerShell 命令包含 `chcp 65001`
   - 章节标题正则支持中文数字

### 2.2 潜在风险 ⚠️

1. **章节解析复杂度被低估**:
   - 凡人修仙传可能有非标准章节格式（番外、卷标题、空行）
   - 中文数字转换需要完整映射表（"第一百二十三章"）
   - 计划中的测试只覆盖简单情况

2. **重叠窗口逻辑易错**:
   - "排除前置重叠章"的实现需要仔细处理边界
   - `commit_chapters` 与 `context_chapters` 的关系容易混淆
   - Hash 计算需要在正确的章节范围上

3. **模板路由过于简单**:
   - 当前设计是字符串匹配（`、` 或 `，` 分割的关键词）
   - 可能误触发（"门派"会匹配到"七玄门"的描述段落）
   - 没有负样本过滤

4. **文档索引搜索有限**:
   - 只按标题搜索，不支持同义词
   - "抽髓丸" vs "抽髓灵丹" 会被识别为不同条目
   - 12 行上下文可能不够判断是否需要回补

5. **DeepSeek 缓存未利用**:
   - 计划中的 LLM 客户端是简单的单轮请求
   - 没有利用 DeepSeek 的前缀缓存特性
   - 重复窗口间的模板文本会重复计费

## 三、与用户需求的差异

### 3.1 用户期望 vs 计划实现

| 需求点 | 用户描述 | MVP 计划 | 差异分析 |
|--------|----------|----------|----------|
| **读取策略** | "1-5, 5-9, 9-13" | 窗口大小5，步长4，`1-5, 5-9, 9-13` | ✅ 一致 |
| **去重逻辑** | "先找是否有重复，完善就补充，完善就跳过" | 文档索引搜索 + 模型判断 | ✅ 符合，但模型判断不可控 |
| **工具调用** | "参考 Reasonix 的已有代码，底层实现直接照搬" | OpenAI HTTP 客户端 | ❌ **不可能照搬 Go 代码** |
| **中断续跑** | "分析过的就不分析了" | SQLite Ledger，hash 判断 | ✅ 一致 |
| **缓存命中** | "调用读写工具，deepseek 缓存命中问题" | 未考虑缓存优化 | ⚠️ 遗漏，可能成本高 |

### 3.2 关键遗漏

1. **DeepSeek 缓存策略未设计**:
   - MVP 每次构建独立 prompt，无法复用前缀
   - 建议：固定 system message，章节和模板以 user message 追加

2. **Token 成本估算缺失**:
   - 凡人修仙传全本约 200 万字
   - 窗口大小 5 章 ≈ 1-2 万字
   - 5 个模板 ≈ 5000 字
   - 每窗口 4 个模板组 × (2万 + 5000) = 10万 tokens/窗口
   - 全书约 400 个窗口 → **4000 万 tokens**
   - 建议计划中增加成本估算和预算控制

3. **错误恢复机制薄弱**:
   - 只有 "mark_failed + 重跑" 一种策略
   - API 限流、超时、模型输出截断都会导致整窗口失败
   - 建议：增加部分成功保存、自动降级窗口大小
