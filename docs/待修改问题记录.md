### 当前默认更新流程约束

当前默认桌面端提取的默认工具链应按下面的方式理解：

- 用 `read_file` 读取当前窗口或允许报告的必要上下文。
- 更新已有报告时，优先用 `read_report_excerpt` 按卡片和字段坐标读取目标字段块，再用 `upsert_report_section` 替换同一字段块。
- `write_file` 只用于创建新报告，或在已经读清楚完整内容、能安全整体重写时使用。
- 当前窗口没有新增信息时，用 `mark_no_update` 标记，不要为了“有输出”而追加空泛内容。
- `edit_file`、`multi_edit`、`grep`、`ls` 不进入默认启用工具链；旧测试或显式 opt-in 场景可以单独启用。

### excerpt/upsert 机制风险点

1. 早期 `read_report_excerpt` 后的写入策略

   旧工具即使命中已有人物小节，也会推荐 `append_to_section`。当前默认链路不能再推荐追加；当命中的是人物卡片这类结构化条目时，应根据模板规则做字段级替换或合并，并提示模型“已有条目应合并整理，不要继续追加同名字段”。

2. 早期 `read_report_excerpt` 的返回算法

   旧返回逻辑是命中后返回整个 Markdown 小节，再按 `maxChars` 截断。对于 `韩立` 这种超长人物小节，这会造成“总是看到前半段，看不到后半段”的问题。如果以后重新设计这类能力，可以考虑：

   - 支持按字段关键词返回字段附近内容，例如只返回 `核心性格`、`代表事件` 等字段及其已有补充。
   - 支持返回小节摘要/字段索引，让模型知道该人物下面已经有多少个同名字段。
   - 对超长小节不要只从开头截断，应围绕命中字段或新信息关键词返回更相关的片段。
   - 返回结果里明确标注“该小节已超长/已截断/存在多个同名字段”，避免模型误判。

3. 早期 `upsert_report_section` 的写入模式

   当时模型实际写入韩立、厉飞雨时使用的是 `append_to_section`。当前默认链路应使用字段级坐标和字段块内容，不应继续使用只能追加、替换整个 section 或追加到末尾的旧写法。

4. 模板/提示词规则

   NPC 性格类模板需要明确：

   - 更新已有人物时，按人物小节合并。
   - 更新已有字段时，按字段语义合并。
   - 禁止在同一人物小节中反复新增同名字段。
   - 禁止用“（补充）”作为长期结构堆积。

5. 输出格式规则

   需要把“正式案例不写来源，待定案例才写来源”写进模板或运行规则里，避免模型默认给所有事件都加章节来源。

6. 验收检查

   后续修复后，至少要用这个报告场景验证：

   - `韩立` 小节中不再出现多条重复的 `核心性格` 字段。
   - 新抽到的韩立性格信息会并入既有 `核心性格` 字段。
   - 正式案例不再统一附带来源。
   - 待定案例仍能保留来源说明。
