# NovelExtractor - 缓存优化功能说明

## ✅ 已实现 DeepSeek 缓存优化

### 功能概述

项目现在支持 **DeepSeek 前缀缓存**，可节省约 **20% 的 token 成本**。

---

## 工作原理

### 传统方式（无缓存）❌
```python
# 每次都发送完整 prompt（~25000 tokens）
messages = [
    {"role": "user", "content": "模板(5000) + 章节(20000)"}
]
# 每次都为全部 25000 tokens 付费
```

### 优化后（有缓存）✅
```python
# 分离可缓存部分和变化部分
messages = [
    {"role": "system", "content": "任务说明 + 模板内容"},  # 5000 tokens (可缓存)
    {"role": "user", "content": "当前窗口章节"}           # 20000 tokens (每次变化)
]
# 第一次：付费 25000 tokens
# 第二次起：付费 20000 tokens (system 部分命中缓存)
```

## 成本对比

### 全书处理（500 窗口 × 4 组）

| 方案 | Token 量 | 成本估算 | 节省 |
|------|---------|---------|------|
| **无缓存** | 50,000,000 | $100-200 | - |
| **有缓存** | 40,000,000 | $80-160 | **20%** ↓ |

### 缓存原理
- **缓存内容**：system message（模板 + 指令）
- **缓存 TTL**：5 分钟
- **命中率**：~20%（模板部分）
- **前提条件**：连续请求的 system message 完全相同

## 技术实现

### 1. Prompt 分离
**文件**：`src/novel_extractor/prompts.py`

```python
# 可缓存部分（固定内容）
def build_system_prompt(template_texts: dict[str, str]) -> str:
    """任务说明 + 模板 + 输出格式"""
    
# 变化部分（每窗口不同）
def build_user_prompt(novel_id, window_id, chapters, ...) -> str:
    """当前窗口的章节内容"""
```

### 2. 客户端支持
**文件**：`src/novel_extractor/llm.py`

```python
class OpenAICompatibleClient:
    def complete_with_cache(self, system_prompt: str, user_prompt: str) -> str:
        """使用 system + user 分离，启用缓存"""
        messages = [
            {"role": "system", "content": system_prompt},  # 可缓存
            {"role": "user", "content": user_prompt}       # 不可缓存
        ]
```

### 3. Pipeline 集成
**文件**：`src/novel_extractor/pipeline.py`

```python
# 自动检测客户端是否支持缓存
if hasattr(llm_client, "complete_with_cache"):
    response = llm_client.complete_with_cache(system_prompt, user_prompt)
else:
    response = llm_client.complete(prompt)  # 降级到传统方式
```

## 最佳实践

### ✅ 推荐做法
1. **保持模板稳定**：处理过程中不要修改模板文件
2. **连续处理**：一次性处理多个窗口，最大化缓存命中
3. **相同配置**：使用相同的模板组设置

### ❌ 避免的情况
1. **频繁修改模板**：会导致 system prompt 变化，缓存失效
2. **间隔太久**：超过 5 分钟缓存过期
3. **修改 temperature**：会导致缓存 key 变化

## FAQ

### Q: 为什么只节省 20%？
**A**: 因为模板内容（~5000 tokens）占总 prompt 的 20%，章节内容（~20000 tokens）占 80%。只有固定的模板部分能被缓存。

### Q: 如何进一步优化？
**A**: 
1. **减少模板大小**：精简模板说明
2. **批量处理**：5 分钟内处理更多窗口
3. **使用更小的窗口**：减少章节 token 量

### Q: 缓存会影响输出质量吗？
**A**: 不会。缓存只是复用计算结果，对模型输出无影响。

### Q: 兼容其他 LLM 吗？
**A**: 是的。代码会自动检测客户端是否支持 `complete_with_cache`，不支持时降级到传统方式。
