# 提取流程大白话梳理

> 本文按当前代码实际行为梳理，重点解释桌面端从“上传小说”到“生成报告”的完整链路。旧设计文档里有些说法仍有参考价值，但只要和当前代码不一致，以当前代码为准。

## 一句话总览

整个提取流程可以理解成：

```text
用户选项目
  -> 配模型、上传小说、选模板、填章节参数
  -> 创建一个提取任务
  -> 点击开始
  -> 主进程按章节切成运行窗口
  -> 为本次任务生成规则快照和窗口文件
  -> 对每个窗口、每批模板调用大模型
  -> 大模型必须通过受限文件工具写 Markdown 报告，或标记本窗口无更新
  -> 主进程登记报告、更新覆盖索引、更新任务进度
  -> 前端收到 jobs:updated，任务卡和资产页刷新
```

换成更白话的说法：程序不会直接把整本书一次性丢给模型，也不会只拿模型最终回复当报告。它会先把书拆成一小段一小段的“窗口”，每个窗口带着本批模板去问模型。模型看到窗口原文后，必须真的调用工具去写或改 `reports` 目录里的 Markdown 文件。写完以后，程序再把这些 Markdown 登记成资产，前端才能在“资产”页看到和预览。

## 0. 桌面端启动后先准备哪些东西

Electron 主进程启动时会创建这些东西：

- `userDataDir`：Electron 的用户数据目录。默认项目列表、供应商配置、模板库都在这里。
- `projectsRoot`：默认项目根目录，默认是 `userDataDir/projects`。
- `credentialsRoot`：API key 保存目录，默认是 `userDataDir/credentials`。
- `settings.json`：保存用户是否改过项目存储目录。
- `providers.json`：保存大模型供应商配置。
- `api-keys.json`：保存 API key。系统支持时会通过 Electron `safeStorage` 加密。

主进程把所有 IPC handler 注册起来，然后创建窗口。前端通过 `window.novelExtractor` 调这些 IPC，不直接读写文件。

## 1. 用户先进入项目

前端如果没有当前项目，会显示项目入口。用户创建或选择项目后，前端会做三件事：

1. 读取模型供应商配置。
2. 读取当前项目能用的模板和模板选择。
3. 读取项目运行态，包括已上传书籍和已有任务。

项目本身会落到项目存储根目录下面。项目运行态在：

```text
<projectRoot>/state/project-runtime.json
```

这个文件里面主要记：

- 上传过哪些书。
- 每本书拆出来的章节记录。
- 创建过哪些任务。
- 已生成哪些报告资产。
- 报告 ID 对应到哪个真实文件路径。

如果程序上次退出时某个任务还标着 `running` 或 `pause_requested`，下次加载时会归一成 `paused`，避免 UI 以为它还在后台继续跑。

## 2. 模型配置怎么进入流程

前端“模型设置”里展示的是 provider store 里已保存且启用的模型，不是直接展示配置里的所有内置预设。

保存供应商时大致是：

```text
用户填 displayName / baseUrl / apiKey / modelName
  -> providers:save
  -> API key 进入 credential store
  -> provider 配置进入 providers.json
```

运行任务时，`windowRunService` 会用：

```text
providerConfigId/modelId
```

拼成模型引用，再通过 provider registry 找到供应商定义、baseUrl、apiKeyRef 和模型名。真正请求时走 OpenAI-compatible 的：

```text
<baseUrl>/chat/completions
```

请求体里会带：

- `model`
- `messages`
- `tools`

返回后统一解析：

- `content`
- `tool_calls`
- `usage`

usage 会被归一成输入 token、输出 token、总 token、缓存命中 token、缓存未命中 token等字段，最后在任务卡里显示成类似：

```text
Token 12345 / 缓存命中率 98.76%
```

## 3. 模板怎么进入流程

模板来源有两类：

- 全局模板：所有项目可用。
- 项目模板：只属于当前项目。

如果 `templates.json` 不存在，主进程会用内置模板初始化。目前默认内置模板包括“丹药分析模板”，默认输出文件名是 `丹药分析.md`。

前端会保存每个项目当前选中的模板 ID。创建任务时，任务只保存这些 `templateIds`，并不把模板正文直接复制进任务记录。

这里有个需要注意的当前实现边界：

- 当前任务记录里保存的是 `templateIds`。
- 点击开始时，主进程会按这些 ID 解析当前模板内容。
- 运行前会生成 `runs/<jobId>/rules/提取规则.md` 规则快照。
- 如果同一个 job 的规则快照已经存在，再次运行会复用旧快照文件。
- 但当前内存里的模板列表仍来自当次解析到的模板对象。

所以它不是严格意义上“创建任务时冻结模板正文”。更准确地说，当前实现是“运行前生成规则快照；同 job 已有快照时复用快照文件”。如果后续要做严格可复现，需要继续核对这块边界。

还有一个硬规则：一次任务里选中的模板不能有重复输出文件名。因为后面工具循环和 outcome 追踪都是围绕 `outputFileName` 来判断这个模板有没有处理完。

## 4. 上传小说发生了什么

前端上传小说支持 `.txt` 和 `.md`，每次只能上传一本。上传时前端会把本地文件路径、项目 ID、显示名发给：

```text
books:uploadTxt
```

主进程做这些事：

1. 校验扩展名只能是 `.txt` 或 `.md`。
2. 读取原文件 buffer。
3. 尝试按 UTF-8 BOM、UTF-8、GBK/CP936 解码。
4. 判断解码后是不是像正常小说文本，比如不能全是乱码、空内容、不可打印字符。
5. 按章节标题切章节。
6. 在项目目录里创建书籍资产目录。
7. 把原文复制为统一的 `source/original.txt`。
8. 把每章正文单独写成 `chapters/0000.txt`、`0001.txt` 这类文件。
9. 保存书籍记录和章节记录到 `project-runtime.json`。

章节标题识别规则很简单，主要认两类：

- 中文：`第 X 章`，可带卷号，可用中文数字或阿拉伯数字。
- 英文：`chapter 1` 这类。

如果找不到任何章节标题，上传会失败。

上传后的典型目录形态：

```text
<projectRoot>/assets/books/<bookId>/
  source/original.txt
  chapters/0000.txt
  chapters/0001.txt
  ...
```

运行时生成窗口时，不是直接用这些章节文件，而是重新读取 `source/original.txt` 再解析章节和切窗口。这些章节文件更像上传资产和运行态记录的一部分。

## 5. 创建任务只是“登记”，还没有开始跑

用户在“提取参数”里选：

- 书籍。
- 模板。
- 单次运行章节数。
- 提取章节窗口。
- 重叠章节数。
- 是否跳过已提取章节。
- 模型。

前端构造 `CreateJobDto` 前会先做一次归一化：

- `singleRunChapterCount` 至少是 1。
- `extractionChapterCount` 至少等于 `singleRunChapterCount`。
- `overlapChapterCount` 最多是 `singleRunChapterCount - 1`。
- 没有书、没有模板、没有模型都会直接报错。

主进程收到 `jobs:create` 后再校验一次，避免只信前端：

- 小说 ID 不能为空。
- 模板数组不能为空。
- providerConfigId 和 modelId 不能为空。
- 章节数必须是正整数。
- 重叠章节数必须是非负整数。
- `extractionChapterCount >= singleRunChapterCount`。
- `overlapChapterCount < singleRunChapterCount`。

然后主进程会估算窗口数量。这个估算只用书籍章节数和窗口参数，不会调用模型。

创建出来的任务初始状态是：

```text
status: created
progressText: 进度：0/<窗口数>
progress: 0/<窗口数>
tokenText: Token 0 / 缓存命中率 0.00%
```

这一步只保存任务记录，不生成窗口文件，不生成规则快照，也不请求大模型。

## 6. 点击开始以后才进入真正运行

用户点“开始”后，前端调用：

```text
jobs:start
```

前端会先乐观地把任务显示成运行中。主进程运行时也会不断发 `jobs:updated` 事件，前端收到后更新任务卡。

当前实现里要注意两点：

1. `jobs:start` 这个 IPC promise 会等整个任务跑完才 resolve，不是马上返回一个后台任务句柄。
2. 但运行过程中主进程会不断推送 `jobs:updated`，所以 UI 仍然能实时刷新。

主进程会用 `activeJobRuns` 防止同一个 job 被重复启动。这个防重是按 jobId 做的，不是全局队列。也就是说当前代码没有一个统一的“所有任务只能同时跑一个”的全局调度器。

启动后，主进程会创建任务日志：

```text
<projectRoot>/runs/<jobId>/logs/<timestamp>.txt
<projectRoot>/runs/<jobId>/logs/<timestamp>.simple.txt
```

完整日志记大模型请求、返回、工具调用、工具返回、错误等详细内容。简化日志给 UI 的“运行流程”面板读，避免直接展示超长完整日志。

## 7. 运行前先生成两个关键产物

真正跑窗口前，`preparePreRunArtifacts` 会准备两类东西。

### 7.1 运行窗口文件

它会读取上传时保存的原文，重新分章，然后根据参数切窗口：

```text
singleRunChapterCount = 每个窗口最多看几章
overlapChapterCount = 下个窗口往前重叠几章当上下文
extractionChapterCount = 本次最多处理前多少章
stride = singleRunChapterCount - overlapChapterCount
```

举个例子：

```text
singleRunChapterCount = 5
overlapChapterCount = 1
extractionChapterCount = 13

窗口 1：上下文 1-5，提交 1-5
窗口 2：上下文 5-9，提交 6-9
窗口 3：上下文 9-13，提交 10-13
```

重叠章节只做上下文，不算本窗口要提交的新章节。

生成后的目录：

```text
<projectRoot>/runs/<jobId>/windows/
  window-0001.txt
  window-0002.txt
  ...
  manifest.json
```

`manifest.json` 里会记录每个窗口的：

- `windowId`
- 序号。
- 文件名。
- 项目相对路径。
- 窗口文本 hash。
- 上下文章节范围。
- 提交章节范围。
- 上下文章节标题。
- 提交章节标题。
- 字符数。

如果已有 manifest 且原文 hash、切分配置 hash、窗口文件 hash 都对得上，就复用已有窗口文件，不重新生成。

### 7.2 提取规则快照

主进程还会生成规则文档：

```text
<projectRoot>/runs/<jobId>/rules/提取规则.md
<projectRoot>/rules/提取规则.md
```

第一个是运行级快照，第二个是项目级 latest copy，方便人工查看最新规则。

规则文档里会写：

- 通用抽取规则。
- 模板路由规则表。
- 模板组快照。
- 每个模板的正文、输出文件名、hash。
- 路由失败策略。
- 写入规则。
- 跳过已提取策略。

注意：当前窗口 prompt 里明确写着“本阶段不做模板路由：当前请求只处理本批次列出的选中模板。”所以当前实际运行不是“先路由命中哪个模板，再只跑命中模板”。规则快照更多承担审计、hash、规则来源和后续扩展的作用。

## 8. 覆盖索引用来判断能不能跳过

运行开始后，`windowRunService` 会加载覆盖索引：

```text
<projectRoot>/metadata/coverage/coverage-index.json
```

覆盖索引记录的是某个“窗口 + 模板 + 规则”的目标是否已经处理过。它的 key 由这些字段组成：

- `bookId`
- `templateId`
- `outputFileName`
- `templateHash`
- `windowHash`
- `rulesSemanticHash`
- `submittedChapterRange`

如果用户勾了“跳过已提取章节”，流程会这样判断：

```text
对每个窗口:
  对每个模板生成 coverage target
  如果 coverage index 已有这个 target:
    这个模板在这个窗口跳过
  如果这个窗口所有模板都跳过:
    整个窗口不请求模型
  如果只跳过一部分模板:
    只把剩下的模板交给模型
```

如果用户点“重新开始”，主进程会用 `skipAlreadyExtracted: false` 跑，也就是不看覆盖索引跳过，尽量全量重跑。

窗口成功处理后，只要某个模板结果是：

- `written`
- `no_update`

就会写入覆盖索引。下次同样窗口、同样模板、同样规则 hash 就能跳过。

## 9. 每个窗口怎么问模型

每个需要处理的窗口，会先读取对应的 `window-000x.txt`。然后按模板分批。当前分批规则非常直接：

```text
每批最多 maxTemplatesPerCall 个模板
默认 maxTemplatesPerCall = 4
```

配置里还有 `promptBudgetChars`，但当前分批实现只按模板数量切，不按 prompt 字符预算切。

对每一个“窗口 + 模板批次”，都会进入一次工具循环。

## 10. 模型收到的 prompt 里有什么

每批工具循环开始时，会构造两条初始消息：

### system message

主要是稳定规则，比如：

- 你是小说资料抽取助手。
- 必须通过当前默认启用的文件工具写入或更新正式模板 Markdown。
- 如果没有新信息，最终文本必须严格返回 `NO_UPDATE`。
- 禁止使用全书先验知识。
- 更新既有报告前，优先用 `read_report_excerpt` 按卡片和字段坐标读取目标字段块，必要时再用 `read_file` 读取允许范围内的窗口或报告上下文。
- 写入既有报告时，优先用 `upsert_report_section` 替换同一字段块，不要把新增信息堆成重复字段。
- `write_file` 只用于创建新报告，或在已经确认可以安全整体重写时使用。
- 如果部分模板无新增，要用 `mark_no_update` 标记。
- `edit_file`、`multi_edit`、`grep`、`ls`、`glob`、`bash` 当前默认禁用，不属于默认提取链路。

### user message

主要是当前窗口和当前批次动态内容：

- 小说名。
- 书籍 ID。
- 当前运行日期。
- 窗口序号。
- 窗口文件项目相对路径。
- 上下文章节范围。
- 提交章节范围。
- 上下文章节标题。
- 提交章节标题。
- 宿主提供的报告清单。
- 当前批次模板的 Prompt Profile。
- 当前窗口文本。

报告清单很关键。主进程会提前告诉模型：

```text
已有报告：xxx.md、yyy.md
待创建报告：zzz.md
不要为了确认这些报告是否存在而重复查目录或搜索；grep、ls、glob 和 bash 当前默认未启用
```

这可以减少模型反复用搜索或列目录工具确认宿主已经给出的信息，也避免把历史上的 `grep` / `ls` / `glob` / `bash` 能力误当成默认可用工具。

模板 Prompt Profile 是从完整模板压缩来的，不是完整模板原文。它会尽量去掉示例、参考、占位内容，保留字段、输出、禁止、规则等核心内容。如果压缩后太短，会使用保守回退卡片，提醒模型只根据当前窗口和已读取报告写。

完整模板仍然会写入规则快照，用于审计和 hash。

## 11. 模型不能只“回答”，必须走工具

模型每轮返回有两种情况。

### 11.1 模型返回工具调用

主进程会逐个执行工具调用，并把工具结果作为 `tool` message 塞回对话，再继续下一轮模型请求。

当前桌面端默认启用的工具来自配置，主要包括：

- 读：`read_file`、`read_report_excerpt`
- 写：`write_file`、`upsert_report_section`
- 状态：`mark_no_update`

这些工具之外，`edit_file`、`multi_edit`、`grep`、`ls`、`glob`、`bash` 默认禁用。因为 `bash` 默认禁用，配套的 `bash_output`、`wait`、`kill_shell` 也不会进入默认启用工具列表。旧实现和 legacy 测试可以显式 opt-in，但正常桌面提取流程不应依赖它们。

不过这些工具都被桌面端套了一层安全边界。模型不是想读哪里就读哪里，想写哪里就写哪里。

### 11.2 模型没有工具调用

如果模型没有工具调用，主进程会看最终文本和 outcome 是否完整：

- 如果最终文本严格等于 `NO_UPDATE`，且本批还没有任何模板 outcome，就把本批所有模板都记为 `no_update`。
- 如果已经每个模板都有 `written` 或 `no_update`，本批完成。
- 如果没有工具、也不是 `NO_UPDATE`，主进程会追加纠正消息，要求模型要么调用写工具，要么严格返回 `NO_UPDATE`。
- 如果只处理了部分模板，主进程会追加纠正消息，只要求补齐缺失的 outputFileName。

所以最终文本不是报告正文。正式报告必须通过工具写到 Markdown 文件里。

## 12. outcome 是怎么判断“本批模板处理完了”

每个批次都会创建一个 `BatchOutcomeTracker`。它盯着本批每个 `outputFileName` 是否已经有结果。

结果有两类算完成：

- `written`：模型通过写工具成功写入或更新了这个报告。
- `no_update`：模型通过 `mark_no_update` 标记这个报告本窗口无新增，或者整批 `NO_UPDATE`。

如果一个批次有 4 个模板，模型只写了其中 1 个报告，就不能结束。主进程会继续追问：

```text
上一轮尚未为本批次所有选中模板提供处理结果，缺少 outputFileName：...
```

这条规则是为了防止多模板同请求时，模型漏掉某个模板。

## 13. 工具读写边界

工具调用进入真正执行前，会经过多层检查。

### 13.1 写工具只能写当前批次允许的报告

写工具的目标文件必须是当前批次模板的 `outputFileName`，不能写别的文件。

允许的写法大致有：

```text
丹药分析.md
reports/丹药分析.md
assets/books/<bookId>/reports/丹药分析.md
```

执行前会归一成裸文件名，然后落到：

```text
<projectRoot>/assets/books/<bookId>/reports/<outputFileName>
```

不允许：

- 绝对路径。
- `..`。
- 子目录。
- 反斜杠或斜杠路径。
- 未选中模板的报告名。

### 13.2 读工具只能读当前窗口和允许报告

默认读工具只能碰当前窗口和当前批次允许的报告：

- 当前窗口文件。
- 当前书籍 reports 目录。
- 当前批次允许的报告文件。

如果模型直接写 `window-0001.txt`，主进程会帮它映射到项目相对路径的窗口文件。

`read_file` 适合读取窗口文本和必要的短报告上下文。`read_report_excerpt` 适合更新旧报告前按 `outputFileName`、卡片名和字段名读取目标字段块；如果上下文不足，应保守处理，不要直接覆盖或追加重复结构。

### 13.3 当前默认的旧报告更新流程

如果目标报告已经存在，模型不能上来就 `write_file` 覆盖。

默认流程应该是：

```text
1. 用 read_report_excerpt 按报告名、卡片名和字段名读取目标字段块。
2. 把当前窗口的新信息和已读字段块合并整理，去重后形成新的字段内容。
3. 用 upsert_report_section 写回同一报告、同一卡片、同一字段。
4. 只有创建新报告，或已完整读清并确认可安全整体重写时，才用 write_file。
5. 本窗口没有新增信息时，用 mark_no_update。
```

对结构化报告来说，重点是“合并整理”而不是“继续追加”。例如已有某个人物小节时，新信息应该并入已有字段或重写相关字段，不应反复新增 `核心性格（补充）` 这类长期堆积字段。

### 13.4 字段级 excerpt/upsert 更新机制

历史上曾经有一条旧版 `read_report_excerpt` -> `upsert_report_section` 更新链路：

```text
read_report_excerpt(...) 先按关键词读相关段落
upsert_report_section(...) 再按 sectionId 追加、替换或追加到末尾
```

当前默认链路保留 `read_report_excerpt` 和 `upsert_report_section`，但语义已经收窄到字段级坐标读写：读取时按报告、卡片和字段拿目标字段块；写入时按同一坐标替换字段块。旧的 section 追加模式不应再作为默认行为，因为它在结构化报告里容易产生糟糕合并效果：长小节会被截断，只看到前半段；命中人物小节后还可能推荐 `append_to_section`；最终模型倾向于不断追加同名字段，而不是字段级归并。

### 13.5 已禁用的搜索、目录、编辑和 shell 能力

`grep`、`ls`、`edit_file`、`multi_edit`、`glob` 和 `bash` 当前默认禁用，正常桌面提取不应调用它们。因为 `bash` 禁用，`bash_output`、`wait`、`kill_shell` 这些后台 shell 配套工具也不会默认启用。

历史上 `bash` 曾经被放在临时 sandbox 里运行：每个窗口复制当前 reports 目录里的普通文件，过滤项目根、reports 根和用户 home 等敏感环境变量，再把满足条件的输出同步回真实 reports。

这段历史机制的安全约束仍有参考价值，但不要把它理解成当前默认能力。以后如果恢复 shell 类工具，至少仍需要保留这些边界：

- 文件名属于当前批次允许 outputFileName。
- 目标是普通文件，不是符号链接或目录。
- 不能删除已有报告。
- 覆盖已有报告前需要先查过相关报告，或者同轮已经写过该报告。

窗口结束后的关闭后台任务、同步报告、删除 sandbox 等逻辑也只属于历史机制或未来重新设计时的参考点。

## 14. 工具错误怎么恢复

工具错误不是都直接让任务失败。很多错误会被包装成“可恢复工具结果”交回给模型，让模型下一轮自己修。

常见可恢复情况：

- 调了未启用工具，比如 `pwd`、`grep`、`ls`、`edit_file`、`multi_edit`、`glob`、`bash`、`bash_output`、`wait`、`kill_shell`。
- 试图用搜索或列目录工具重复确认报告清单已经提供的存在性信息。
- `read_file` 或 `read_report_excerpt` 读了不允许的路径或报告。
- `upsert_report_section` 的字段坐标或 `updates` 参数结构不对。
- 工具参数结构不对。
- 更新旧报告时没有先定位和读取足够上下文，就试图直接覆盖或追加。
- 写报告内容里出现内部路径或运行窗口文件名。
- 报告里出现 `状态：模板` 或 `状态：草案`。

每类错误会给模型一个 hint。比如更新旧报告上下文不足时，会提示：

```text
先用 read_report_excerpt 按报告、卡片和字段读取目标字段块；整理新旧内容后，用 upsert_report_section 写回同一字段。只有创建新报告或确认可安全整体重写时，才使用 write_file。
```

同一个可恢复工具错误如果重复超过配置次数，任务会失败。默认重复上限是 3。

## 15. 报告内容还有额外防线

写报告前，主进程会检查写入内容片段，避免把内部运行信息写进正式 Markdown。

会拦的内容包括：

- `runs/job...`
- `runs/<jobId>/windows`
- `assets/books`
- Windows 绝对路径。
- AppData 项目路径。
- “后续窗口”这类流程性说法。
- `window-0001` 这种内部窗口文件名。
- `状态：草案` 或 `状态：模板`。

如果命中，工具不会真的写入，而是把错误和修改建议返回给模型，让模型改写成用户能看的来源描述，比如“窗口 1（第1-3章）”或章节名。

## 16. 报告写入后怎么登记成资产

每当写工具成功写入报告，`windowRunService` 会调用 `registerTemplateOutputReport`。

它会：

1. 找到真实报告路径：

   ```text
   <projectRoot>/assets/books/<bookId>/reports/<outputFileName>
   ```

2. 读取文件大小。
3. 如果以前登记过同名报告，复用旧 report ID。
4. 如果是新报告，生成新 report ID。
5. 记录：

   - `bookId`
   - `fileName`
   - `displayName`
   - `relativePath`
   - `byteSize`
   - `createdAt`
   - `updatedAt`
   - `reportKind: template-output`

6. 保存到 `project-runtime.json`。

资产页读取报告列表时走 `books:listReports`，按更新时间倒序返回。点某个报告时走 `reports:preview`，主进程读 Markdown，用安全 Markdown renderer 转 HTML 和标题索引后返回前端。

当前配置里有 `rawWindowReportDefaults`，类型里也保留 `raw-window`，但当前主流程不会生成 raw-window 报告。正常提取产物是 `template-output`。

## 17. 任务进度怎么算

底层 `createJobRuntime` 按窗口循环：

```text
job.started
  -> window.started
  -> llm.completeWindow
  -> window.completed
  -> 下一个窗口
  -> job.completed 或 job.failed
```

每个窗口完成后会累加：

- completedWindowCount
- skippedWindowCount
- executedWindowCount
- executedWindowElapsedMs
- usage
- fee

当前工具费用固定为 0 CNY，UI 主要展示 token 和缓存命中率。

主进程收到 runtime state 后，会转成任务记录补丁：

- `status`
- `progressText`
- `tokenText`
- `failureReason`
- `progress`
- `timing`

然后保存任务、推送 `jobs:updated` 给所有窗口。

预计总耗时有两种来源：

- 还没有完成窗口时，用 150 到 190 秒之间的初始窗口估算乘以有效窗口数。
- 已经执行过窗口后，用已执行窗口平均耗时估算。

如果窗口被覆盖索引跳过，它会进入 skipped 计数，不算执行耗时。

## 18. 日志分完整日志和简化日志

每个任务开始时创建两份日志：

```text
runs/<jobId>/logs/<timestamp>.txt
runs/<jobId>/logs/<timestamp>.simple.txt
```

完整日志会写：

- 构建信息。
- 任务信息。
- 任务上下文。
- 窗口上下文。
- 覆盖索引预检。
- 每轮大模型请求。
- 每轮大模型返回。
- 每次工具调用。
- 每次工具返回。
- 可恢复错误原因。
- 批次结果。
- 覆盖索引更新。
- 窗口或任务错误。

为了避免完整日志巨大，模型请求日志会保留 messages 结构，但把窗口正文替换成：

```text
[窗口原文见 window-0001.txt]
```

工具 schema 日志只保留工具名和参数，不铺开完整描述。

简化日志通过 `taskProgressLog` 把详细事件压成适合 UI 展示的文字。前端任务卡里的“运行流程”读的是简化日志；“打开完整日志”才打开完整 txt。

## 19. 暂停、继续、重新开始、删除当前实际行为

这部分容易误会，要按当前桌面主进程代码讲。

### 暂停

前端运行中任务会显示“暂停”按钮。点击后调用：

```text
jobs:pause
```

当前 `p0Handlers.ts` 的处理是把任务状态更新为：

```text
status: paused
progressText: 已暂停
```

但它没有把暂停信号传给正在执行的 `createJobRuntime`。底层 runtime 本身有“窗口边界暂停”的能力，不过桌面这层当前没有接进去。

所以当前实现里，暂停更像“更新任务记录/UI 状态”，不是可靠的强中断。正在跑的模型请求和窗口流程可能继续，并在后续 runtime state 更新时覆盖任务状态。

### 继续

点击继续会调用：

```text
jobs:resume
```

当前实现是重新进入 `runOrReuseModelBackedJob`。如果同一个 job 正在 active run，就复用那条 promise；如果没有，就重新跑窗口流程。

因为默认 `skipAlreadyExtracted` 为 true，所以继续通常依赖覆盖索引跳过已经处理过的目标，而不是从某个内存断点精准续跑。

### 重新开始

点击重新开始会调用：

```text
jobs:restart
```

它会用：

```text
skipAlreadyExtracted: false
```

也就是不看覆盖索引跳过，尽量重新处理所有窗口和模板。

### 删除任务

删除任务只移除任务记录。主进程实际做的是从运行态 `jobs` 列表里删掉这个 job，不会顺手删除共享 Markdown 报告，也不会把这个动作当成一次报告文件清理。

UI 文案也写了：

```text
确认后只移除任务记录和运行日志索引。共享报告文件不在本次操作范围内。
```

## 20. 失败会在哪里发生

常见失败点大概有这些：

### 上传阶段

- 文件不是 `.txt` 或 `.md`。
- 编码不是可识别的 UTF-8/GBK 类小说文本。
- 文本为空或像乱码。
- 找不到章节标题。
- 写入书籍资产目录时路径冲突。

### 创建任务阶段

- 没有选择小说。
- 没有选择模板。
- 没有选择模型。
- 章节参数不合法。
- 模板输出文件名重复。
- 任务 ID 或运行目录冲突。

### 运行前准备阶段

- 原文路径不在项目目录内。
- 原文重新解析没有章节。
- 窗口参数非法。
- 规则快照生成失败。
- 规则快照已存在但读取/复制 latest 失败。

### 模型阶段

- provider 没配置。
- model 不属于 provider。
- API key 不存在。
- HTTP 请求失败。
- 返回格式异常，无法得到有效工具调用或内容。

### 工具阶段

- 路径越界。
- 写了未选中的报告。
- 旧报告未查询就覆盖。
- 读取超大旧报告。
- 历史/遗留 `upsert_report_section` 链路里才会出现的 Markdown sectionId 找不到（当前默认流程不依赖 sectionId）。
- 同一个可恢复错误重复太多次。
- 写入内容包含内部路径、窗口文件名或模板/草案状态。

失败后，任务会更新为 `failed`，`failureReason` 会显示到任务卡，完整日志里会有更详细的窗口、轮次和工具信息。

## 21. 输出目录结构速查

一个项目跑起来后，核心目录大概长这样：

```text
<projectRoot>/
  state/
    project-runtime.json

  assets/
    books/
      <bookId>/
        source/
          original.txt
        chapters/
          0000.txt
          0001.txt
        reports/
          丹药分析.md
          其他模板输出.md

  runs/
    <jobId>/
      windows/
        window-0001.txt
        window-0002.txt
        manifest.json
      rules/
        提取规则.md
      logs/
        20260704-120000.txt
        20260704-120000.simple.txt

  rules/
    提取规则.md

  metadata/
    coverage/
      coverage-index.json
```

如果要人工排查：

1. 先看任务卡失败原因。
2. 再点“运行流程”看简化日志。
3. 还不够就打开完整日志。
4. 对照 `runs/<jobId>/windows/manifest.json` 看窗口范围。
5. 对照 `assets/books/<bookId>/reports` 看真实报告。
6. 对照 `metadata/coverage/coverage-index.json` 看哪些窗口模板目标被认为已覆盖。

## 22. 当前实现和名字上“队列”的关系

前端页面叫“提取任务”，也有任务列表筛选，但当前代码不是一个自动排队系统。

创建任务只是把任务放进列表。用户点击开始，那个任务才跑。代码里只防止同一个 job 被重复启动，并没有把所有 job 放进一个统一队列里串行消费。

如果未来要做真正队列，需要单独设计：

- 全局并发数。
- 等待队列。
- 取消/暂停信号。
- 任务优先级。
- UI 上 pending/running 的语义。

## 23. 当前最值得记住的几个边界

- 当前运行不做模板路由，处理的是本批次列出的选中模板。
- 创建任务不等于开始提取，开始时才生成窗口和规则。
- `jobs:start` 当前会等任务结束才 resolve，但过程中会发 `jobs:updated`。
- 暂停按钮当前没有接到底层 runtime 暂停信号，不能当强中断。
- 继续通常靠覆盖索引跳过已完成目标，不是精确内存断点续跑。
- 重新开始会关闭跳过逻辑，尽量重跑所有窗口模板。
- 模型最终回复不是报告，报告必须通过工具写入 Markdown。
- 旧报告更新前优先用 `read_report_excerpt` 按卡片和字段读取目标字段块，再用 `upsert_report_section` 写回合并后的字段内容；`write_file` 只适合创建或安全整体重写。
- 写工具只能写当前批次模板允许的 outputFileName。
- 正式报告里不能出现内部路径、窗口文件名、模板/草案状态。
- 覆盖索引的 key 包含模板 hash、窗口 hash 和规则 hash，模板或规则变了就不会命中旧覆盖。

## 24. 如果要继续梳理，可以从这些问题往下拆

这份文档只是把当前流程摊开。后续如果要继续优化，可以优先看这些问题：

1. 暂停/继续是否要接入底层 runtime 的窗口边界暂停。
2. `jobs:start` 是否应该真正后台化，立即返回启动状态。
3. 是否需要全局任务队列，避免多个 job 同时跑。
4. 模板快照是否要在创建任务时冻结，保证重跑完全可复现。
5. 当前“不做模板路由”的策略是否就是最终策略，还是要恢复路由层。
6. `promptBudgetChars` 是否要真正接入模板分批。
7. 覆盖索引损坏策略是否应该从 fail 改成 conservative-rerun。
8. 日志中的关键字段是否足够支持用户排查真实失败。
