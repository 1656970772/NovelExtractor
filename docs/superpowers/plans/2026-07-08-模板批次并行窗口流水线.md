# 模板批次并行窗口流水线实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task by task. 每个任务先写 RED 测试，再做最小实现，再运行本任务列出的验证命令。

## 目标

将桌面端提取任务从“窗口串行、窗口内模板批次串行”改成“模板批次并行流水线”：按 `templateBatchSize` 严格顺序固定大小切批，每个批次独立按窗口推进；单个批次遇到普通 LLM/provider 失败时等待 `failureRetryIntervalMs` 后从失败窗口重试，其他批次继续。

## 架构边界

- 不修改 `packages/jobs`。并行批次运行只在 `apps/desktop/src/main/windowRunService.ts` 层引入适配器。
- 必须保留 `createJobRuntime` 暴露给外部的 `JobRuntime` 语义：`startJob`、`pauseJob`、`resumeJob`、`cancelJob`、`deleteJob`、`getJobState` 和 `events` 对调用方仍可用。
- 并行实现可以在 `windowRunService` 内提供等价 `JobRuntime` facade，并且必须通过现有 `onRuntimeCreated({ jobId, runtime })` 注册；`p0Handlers` 的 pause/resume/cancel/delete 入口不得绕开 runtime 注册表。
- `CreateJobDto` 新增的 `templateBatchSize` 是新任务必填字段；旧持久化任务允许缺省并在读取时补默认值。
- 每批次必须使用独立 `RuntimeChatClient` 与独立 fallback state，避免自动模型 fallback 在批次之间串扰。
- 每批次使用独立详细日志与独立 batch coverage index，但不能破坏跨 job 的 `skipAlreadyExtracted`。保留现有全局 coverage index 作为聚合索引和兼容读取源；每批窗口成功后同时写全局索引与批次索引。恢复旧任务或跨 job 跳过时先读取全局索引，批次索引用于本 job 的批次日志、恢复诊断和局部校验。
- 执行阶段提交检查点是可选动作，只有用户明确要求提交时才执行；本计划不要求暂存或提交。

## 命令规则

- 真实可执行命令统一加 `rtk`。
- 配置包测试在配置包内传 `src/...` 路径，例如：
  - `rtk pnpm --filter @novel-extractor/config test src/defaults.test.ts`
  - `rtk pnpm --filter @novel-extractor/config test src/configInvariants.test.ts`
- desktop 测试在 desktop 包内传 `src/main/...test.ts` 或 `src/renderer/...test.tsx` 路径，例如：
  - `rtk pnpm --filter @novel-extractor/desktop test src/main/windowRunService.test.ts`
  - `rtk pnpm --filter @novel-extractor/desktop test src/renderer/features/extraction/extractionViewModel.test.ts`
- 不写 workspace 根路径测试参数，不在 `pnpm ... test` 后额外加中间的 `--`。

```ts
async function tryAllocateLogPath(input: {
  absolutePath: string;
  relativePath: string;
}): Promise<{ absolutePath: string; relativePath: string } | null> {
  try {
    const handle = await fs.open(input.absolutePath, "wx");
    await handle.close();
    return input;
  } catch (error) {
    const code = (error as NodeJS.ErrnoException).code;
    if (code === "ENOENT") {
      await fs.mkdir(path.dirname(input.absolutePath), { recursive: true });
      try {
        const handle = await fs.open(input.absolutePath, "wx");
        await handle.close();
        return input;
      } catch (retryError) {
        const retryCode = (retryError as NodeJS.ErrnoException).code;
        if (retryCode === "EEXIST") {
          return null;
        }
        throw retryError;
      }
    }
    if (code === "EEXIST") {
      return null;
    }
    throw error;
  }
}

async function allocateLogPath(input: {
  baseFileName: string;
  jobId: string;
  projectRoot: string;
  logDirectorySegments?: readonly string[];
}): Promise<{ absolutePath: string; relativePath: string }> {
  const directorySegments = input.logDirectorySegments ?? ["runs", input.jobId, "logs"];
  for (let index = 0; index < 1000; index += 1) {
    const suffix = index === 0 ? "" : `-${String(index).padStart(3, "0")}`;
    const fileName = `${input.baseFileName}${suffix}.txt`;
    const relativePath = toProjectRelativePath(...directorySegments, fileName);
    const absolutePath = path.join(input.projectRoot, ...directorySegments, fileName);
    const allocated = await tryAllocateLogPath({ absolutePath, relativePath });
    if (allocated) {
      return allocated;
    }
  }
  throw new Error("无法创建唯一任务日志文件");
}
```

  sleep(ms: number): Promise<"elapsed" | "woken">;
  waitForRunPermission?(): Promise<void>;

  onProgress(progress: {

  }): Promise<void> | void;
  onBatchFailure(input: {
    batch: TBatch;
    window: TWindow;
    windowIndex: number;
    error: unknown;
    nextRetryDelayMs: number;
  }): Promise<void> | void;
}
```

实现要点：

- 每个批次独立 `for window`。
- `TBatch` 不要求 `templateCount`，coordinator 内不得缓存、推导或读取批大小；所有目标计数只能来自 `processWindow` 的返回值。
- coordinator 顶层维护共享 `let abortedError: TemplateBatchPipelineAbortError | null = null`。首个 worker 捕获 `TemplateBatchPipelineAbortError` 时只写入第一次错误；后续 worker 读取到同一个 `abortedError` 后停在最近的 abort 边界，不覆盖首个错误。

- 普通失败 sleep 前、sleep 后都检查共享 abort；sleep 返回 `"woken"` 时也必须进入 `waitForRunPermission`，由 facade 处理 pause/resume，并在 pause gate 返回后再次检查共享 abort。

- 进度只使用 `TemplateBatchPipelineProcessResult` 返回的三个目标计数累加：`completedTemplateTargetCount`、`skippedTemplateTargetCount`、`executedTemplateTargetCount`；不要再用旧的布尔跳过字段推导进度。RED 测试要断言 `onProgress` 收到三类计数和 `usage`，覆盖批大小大于 1 且部分模板跳过时 ETA 分母/分子仍准确。

  jobId: string;

  totalTemplateTargetCount: number;

  skippedTemplateTargetCount: number;
  executedTemplateTargetCount: number;
  executedWindowElapsedMs: number;

  fee: FeeAmount | null;
  failureReason?: string;
  pauseRequested: boolean;
  cancelRequested: boolean;
  deleteRequested: boolean;
}

function createParallelRuntimeStatePatch(state: ParallelTemplateBatchRuntimeState): JobRuntimeState {
  return {
    jobId: state.jobId,
    status: state.status,
    completedWindowCount: state.completedTemplateTargetCount,
    totalWindowCount: state.totalTemplateTargetCount,
    skippedWindowCount: state.skippedTemplateTargetCount,
    executedWindowCount: state.executedTemplateTargetCount,
    executedWindowElapsedMs: state.executedWindowElapsedMs,

    fee: state.fee,

}
```

- `createParallelRuntimeStatePatch` 与 `getJobState` 都必须返回 clone：`usage` 至少浅拷贝，`fee` 如果是对象也要浅拷贝；测试通过修改返回对象确认不会污染内部 state。

- `startJob` catch/abort 到 `JobRuntimeResult` / `JobRuntimeError` 的转换必须收口：`TemplateBatchPipelineAbortError` code 为 `"cancelled"` 时保存/发布 cancelled 并返回 `{ ok: true, state }`；code 为 `"deleted"` 时保存/发布 deleted 并返回 `{ ok: true, state }`；code 为 `"fatal_config"` 或普通 `Error` 时保存/发布 failed 并返回 `{ ok: false, error: { code: "job_failed", message } }`；非 `Error` 值也必须用 `toErrorMessage` 等价逻辑转成 message。

- `pauseJob`、`resumeJob`、`cancelJob`、`deleteJob` 的无 job 和非法状态返回值沿用 `JobRuntimeError`：`job_not_found` 或 `invalid_job_state`，不得 throw 给调用方。
- 每次批次窗口完成后累加三个计数字段并调用 `saveState()`；`saveState()` 内部等价于 `options.onRuntimeState(createParallelRuntimeStatePatch(state))`。
- `p0Handlers.toJobPatchFromRuntimeState` 继续接收标准 `JobRuntimeState`，不直接依赖并行内部状态。
- `totalTemplateTargetCount = runtimeWindowManifest.windows.length * reportArtifacts.templates.length`，`batchTotal = templateBatches.length`，不得写 0。
- `pauseJob` 设置 `pauseRequested` 并把状态置为 `pause_requested`；协调器在每个 batch 窗口边界调用 facade 的 pause gate。所有活跃窗口完成且没有新窗口启动后进入 `paused`。

- `resumeJob` 解除 pause gate 并恢复。
- facade 内部维护一个 `wakeControlWaiters()` 等价函数，统一唤醒 retry sleep、pause gate 和其他控制等待者；该函数只负责唤醒，不直接保存 terminal state。
- `cancelJob` 设置 `cancelRequested`，调用 `wakeControlWaiters()`，`shouldAbort` 返回 `TemplateBatchPipelineAbortError("Job cancelled", "cancelled")`。
- `deleteJob` 设置 `deleteRequested`，调用 `wakeControlWaiters()`，`shouldAbort` 返回 `TemplateBatchPipelineAbortError("Job deleted", "deleted")`，并保持现有删除报告/状态语义。
- `fatal_config` 必须走同一条 control wake 路径：`processWindow` 或其 wrapper 在把配置错误转换成 `TemplateBatchPipelineAbortError("...", "fatal_config")` 前先调用 `wakeControlWaiters()`，让其他批次从 sleep/pause gate 醒来后读取 coordinator 的共享 `abortedError` 并退出。
- `runJobWindows` 中 facade 创建后必须先 `await options.onRuntimeCreated?.({ jobId: job.id, runtime })`，再调用 `runtime.startJob(...)`；`startJob` settle 后必须在 `finally` 中 `await options.onRuntimeSettled?.({ jobId: job.id, runtime })`。两个生命周期回调都只调用一次，且 `onRuntimeSettled` 必须等待 `startJob` 完成 coordinator drain、terminal state 映射、terminal state 保存和 terminal event 发布后再执行。

- [ ] **Step 6: 提取单批次单窗口执行函数**

先从当前 `runManifestWindow` 的 outcome coverage 写入段提取明确 helper，不保留“复用现有逻辑”占位。helper 负责把 `executeWindowToolLoop` 返回的每个 `written/no_update` outcome 转成 coverage record，并同时写入全局聚合 index 与当前 batch index。

```ts
type WindowToolLoopOutcome = Awaited<ReturnType<typeof executeWindowToolLoop>>["outcomes"][number];

async function recordBatchCoverageOutcome(input: {
  artifacts: WindowRunArtifacts;
  batchCoverageIndex: ReportCoverageIndexStore;
  clock: { now(): string };
  globalCoverageIndex: ReportCoverageIndexStore;
  manifestWindow: RuntimeWindowManifestWindow;
  outcome: WindowToolLoopOutcome;
  rulesSemanticHashByTemplateId: Map<string, string>;
  templatesByOutputFileName: Map<string, FormalReportTemplate>;
}): Promise<{
  recorded: boolean;
  outputFileName: string;
}> {
  if (input.outcome.status !== "written" && input.outcome.status !== "no_update") {
    return { recorded: false, outputFileName: input.outcome.outputFileName };
  }

  const template = input.templatesByOutputFileName.get(input.outcome.outputFileName);
  if (!template) {
    return { recorded: false, outputFileName: input.outcome.outputFileName };
  }

  const rulesSemanticHash = input.rulesSemanticHashByTemplateId.get(template.id);
  if (!rulesSemanticHash) {
    throw new Error(`Missing rules semantic hash for template ${template.id}`);
  }

  const record = {
    ...createCoverageTarget({
      artifacts: input.artifacts,
      manifestWindow: input.manifestWindow,
      rulesSemanticHash,
      template
    }),
    status: input.outcome.status,
    updatedAt: input.clock.now()
  };
  input.globalCoverageIndex.recordCovered(record);
  input.batchCoverageIndex.recordCovered(record);
  return { recorded: true, outputFileName: input.outcome.outputFileName };
}
```

`runManifestWindowTemplateBatch` 入参必须包含当前批次的 artifacts、logger、chat client、context、`batchTotal`，其中 `context` 同时持有 `globalCoverageIndex` 和当前批次 `batchCoverageIndex`。该函数的 coverage 判断必须先读取全局 coverage plan，再与 `input.batch.templates` 求交：全批 covered 时不发 LLM 请求；部分 covered 时只把未 covered 模板传给 `executeWindowToolLoop`；不得固定使用 `input.batch.templates` 全量执行。

```ts
async function runManifestWindowTemplateBatch(input: {
  artifacts: WindowRunArtifacts;
  batch: TemplateBatch<FormalReportTemplate>;
  batchLogger: TaskTextLogger;
  batchTotal: number;
  chatClient: RuntimeChatClient;
  context: WindowRunJobContext;
  clock: { now(): string };
  job: WindowRunJobInput;
  manifestWindow: RuntimeWindowManifestWindow;
  totalWindowCount: number;
}): Promise<{
  usage: TokenUsage;
  completedTemplateTargetCount: number;
  skippedTemplateTargetCount: number;
  executedTemplateTargetCount: number;
}> {
  const coveragePlan = input.context.coveragePlanByWindowId.get(input.manifestWindow.windowId);
  if (!coveragePlan) {
    throw new Error(`Missing coverage plan for window ${input.manifestWindow.windowId}`);
  }

  const templatesToProcessById = new Set(
    coveragePlan.templatesToProcess.map((template) => template.id)
  );
  const templatesToExecute = input.batch.templates.filter((template) =>
    templatesToProcessById.has(template.id)
  );
  const completedTemplateTargetCount = input.batch.templates.length;
  const executedTemplateTargetCount = templatesToExecute.length;
  const skippedTemplateTargetCount =
    completedTemplateTargetCount - executedTemplateTargetCount;

  if (templatesToExecute.length === 0) {
    await input.batchLogger.append(["上下文", "覆盖索引跳过窗口"], {
      批次: input.batch.batchId,
      窗口: `${input.manifestWindow.index + 1}/${input.totalWindowCount}`,
      窗口文件: input.manifestWindow.fileName,
      当前批次模板数: completedTemplateTargetCount,
      已覆盖模板数: skippedTemplateTargetCount,
      实际执行模板数: 0
    });
    return {
      usage: { ...EMPTY_USAGE },
      completedTemplateTargetCount,
      skippedTemplateTargetCount,
      executedTemplateTargetCount
    };
  }

  if (skippedTemplateTargetCount > 0) {
    await input.batchLogger.append(["上下文", "覆盖索引跳过部分模板"], {
      批次: input.batch.batchId,
      窗口: `${input.manifestWindow.index + 1}/${input.totalWindowCount}`,
      当前批次模板数: completedTemplateTargetCount,
      已覆盖模板数: skippedTemplateTargetCount,
      实际执行模板数: executedTemplateTargetCount
    });
  }

  const windowText = await readWindowText(input.artifacts.project, input.manifestWindow);
  const toolLoopRoundReasonCounts = new Map<ToolLoopRoundReason, number>();
  const result = await (async (): Promise<Awaited<ReturnType<typeof executeWindowToolLoop>>> => {
    try {
      return await executeWindowToolLoop({
        artifacts: {
          ...input.artifacts,
          templates: templatesToExecute
        },
        batchIndex: input.batch.batchIndex,
        batchMaxTemplatesPerCall: input.job.input.templateBatchSize,
        batchTotal: input.batchTotal,
        chatClient: input.chatClient,
        job: input.job,
        manifestWindow: input.manifestWindow,
        taskLogger: input.batchLogger,
        totalWindowCount: input.totalWindowCount,
        toolLoopRoundReasonCounts,
        windowText
      });
    } finally {
      await input.batchLogger.append(["调试", "工具循环轮次原因汇总"], {
        批次: input.batch.batchId,
        窗口: `${input.manifestWindow.index + 1}/${input.totalWindowCount}`,
        汇总: Object.fromEntries(toolLoopRoundReasonCounts)
      });
    }
  })();

  const templatesByOutputFileName = new Map(
    templatesToExecute.map((template) => [template.fileName, template])
  );
  let coverageRecordCount = 0;
  for (const outcome of result.outcomes) {
    const recordResult = await recordBatchCoverageOutcome({
      artifacts: input.artifacts,
      batchCoverageIndex: input.context.batchCoverageIndex,
      clock: input.clock,
      globalCoverageIndex: input.context.globalCoverageIndex,
      manifestWindow: input.manifestWindow,
      outcome,
      rulesSemanticHashByTemplateId: input.context.rulesSemanticHashByTemplateId,
      templatesByOutputFileName
    });
    if (recordResult.recorded) {
      coverageRecordCount += 1;
    }
  }

  if (coverageRecordCount > 0) {
    await input.context.globalCoverageIndex.save();
    await input.context.batchCoverageIndex.save();
    await input.batchLogger.append(["上下文", "覆盖索引更新"], {
      批次: input.batch.batchId,
      全局索引路径: COVERAGE_INDEX_DEFAULTS.relativePath,
      批次索引路径: toTemplateBatchCoverageRelativePath(input.job.id, input.batch.batchId),
      窗口: `${input.manifestWindow.index + 1}/${input.totalWindowCount}`,
      新增或更新记录数: coverageRecordCount
    });
  }

  return {
    usage: result.usage,
    completedTemplateTargetCount,
    skippedTemplateTargetCount,
    executedTemplateTargetCount
  };
}
```

record/save 要点：

- coverage 跳过判断读取全局聚合 index；当前 batch index 只作为本 job/batch 的诊断和恢复证据，不替代全局 skip 源。
- 对每个窗口和批次，先用全局 coverage plan 的 `templatesToProcess` 与 `input.batch.templates` 求交，`completedTemplateTargetCount = input.batch.templates.length`，`skippedTemplateTargetCount = input.batch.templates.length - templatesToExecute.length`，`executedTemplateTargetCount = templatesToExecute.length`。
- `executeWindowToolLoop` 只接收 `templatesToExecute`；全批 covered 时直接返回完整 `EMPTY_USAGE` 和 `executedTemplateTargetCount: 0`，不发 LLM 请求。
- `executeWindowToolLoop` 入参必须扩展为可接收 `taskLogger` 或 `batchLogger`，并把 model request、provider body、tool loop、retry、fallback、tool error 等详细日志全部写入当前批次日志；主 logger 只保留任务摘要、批次失败摘要和全局状态摘要。
- `recordBatchCoverageOutcome` 对同一个 record 同时调用 `globalCoverageIndex.recordCovered(record)` 与 `batchCoverageIndex.recordCovered(record)`。
- 只有 `coverageRecordCount > 0` 时才保存两个 index；保存顺序为全局 index 后 batch index，并在 `batchLogger` 记录两条路径。
- 跳过窗口返回的 usage 使用完整 `EMPTY_USAGE`。`executeWindowToolLoop` 调用必须传真实 `batchTotal: templateBatches.length`。

- [ ] **Step 7: 接入 coordinator**

`runJobWindows` 流程：

1. `planTemplateBatches({ templates: reportArtifacts.templates, maxTemplatesPerCall: job.input.templateBatchSize })`。
2. 加载现有全局 coverage index，作为跨 job skipAlreadyExtracted 的聚合读取源。
3. 为每批创建 artifacts、logger、batch context、chat client；batch context 同时持有全局 coverage index 引用和当前 batch coverage index。
4. 创建 parallel runtime facade 并 `await options.onRuntimeCreated?.({ jobId: job.id, runtime })`。
5. facade 的 `startJob` 调用 `runTemplateBatchPipelines`，并等待 coordinator `Promise.allSettled` drain 完成后才映射 terminal state。
6. 传给 coordinator 的 batch 只需要 `batchId`、`batchIndex` 以及 `TemplateBatch<FormalReportTemplate>` 自身已有的 `templates`；不要新增或透传 `templateCount`。
7. `processWindow` 内用 `requireBatchValue` 获取批次依赖，并由 `runManifestWindowTemplateBatch` 根据当前窗口和当前批次实际执行结果返回计数。
8. `processWindow` 返回 `completedTemplateTargetCount`、`skippedTemplateTargetCount`、`executedTemplateTargetCount`、`usage`；`onProgress` 按这些字段累加 facade state，不再接收或推导旧的布尔跳过字段，也不读取 batch 上的模板数量字段。
9. `onProgress` 或窗口成功路径保存全局 coverage index 与 batch coverage index。
10. `onBatchFailure` 写当前批次详细失败日志；主 logger 只写批次失败摘要和下一次重试时间。
11. cancel/delete/fatal_config 触发时调用 facade 的 `wakeControlWaiters()`，让 sleep/pause gate 立即醒来并通过 coordinator 共享 abort 停止进入新窗口。
12. `finally` 调用 `onRuntimeSettled`，但只能发生在 `startJob` 已经完成 drain 和 terminal state 保存之后。

`startJob` 内调用 coordinator 时必须显式传入批次重试间隔，等价于：

```ts
const DEFAULT_CONFIG = getDefaultConfig();

await runTemplateBatchPipelines({
  batches,
  windows,
  retryIntervalMs:
    options.templateBatchFailureRetryIntervalMs ??
    DEFAULT_CONFIG.extractionRuleDefaults.templateBatching.failureRetryIntervalMs,
  sleep,
  waitForRunPermission,
  shouldAbort,
  processWindow,
  onProgress,
  onBatchFailure
});
```

这里的 `retryIntervalMs` 严格对应 `extractionRuleDefaults.templateBatching.failureRetryIntervalMs`，不得引用 `jobFailureRetryDefaults`、p0Handlers 的 job-level `failureRetryIntervalMs` 或 failure retry scheduler 的默认值。

"jobs:delete": async (input) => {
  if (!input.confirm) {
    throw new Error("Delete confirmation is required");
  }
