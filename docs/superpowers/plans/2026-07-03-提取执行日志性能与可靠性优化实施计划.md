# 提取执行日志性能与可靠性优化 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 降低桌面端提取任务在覆盖索引、工具循环、prompt 日志、模板批处理和报告编辑上的重复成本，并让失败可恢复、可定位、可重跑。

**Architecture:** 将本次日志暴露的问题拆成四层处理：job 级运行上下文缓存覆盖索引与规则 hash，窗口级日志明确记录跳过与处理状态，工具层消除报告查找、旧报告读取和 Markdown 写入导致的多轮根因，日志层保留完整 messages 的可审计性但把窗口原文替换为窗口文件引用、把工具说明压缩为名称和参数。所有阈值、策略和日志级别走配置或集中常量，不把本次 `job-9`、`book-12`、具体小说名或固定模板列表写入生产逻辑。

**Tech Stack:** Electron main process、TypeScript、Vitest、Node `fs/promises`、现有 Reasonix 工具、`packages/config` 默认配置与 invariant、桌面端任务日志文件。

## 背景证据

本计划基于日志 `C:/Users/Administrator/AppData/Roaming/@novel-extractor/desktop/projects/project-1w0t5sx/runs/job-9/logs/20260702-075238.txt` 与当前代码审查整理。

关键现象：

- 完整日志 54,718 行，约 5.3 MB；其中 `[大模型请求][Prompt]` 43 次，prompt 日志累计约 256 万字符。
- 前 6 个窗口覆盖索引全命中，但每个窗口仍重复输出覆盖索引详情。
- 实际处理窗口 7、8、9；窗口 8 运行 22 轮，约 62.9 万 token。
- 窗口 9 因 `Tool is not enabled: pwd` 失败；当前源码已有 unknown-tool 可恢复逻辑与测试，说明运行包版本需要记录和核对。
- `glob **/报告名.md` 每个处理窗口都会触发安全拒绝，累计 12 次 `UNSAFE_PATH`。
- `edit_file/multi_edit` 对长 Markdown 使用自由字符串锚点，出现 `old_string is not unique`、`old_string not found` 和 JSON 参数解析失败。
- `packages/config/src/defaults.ts` 已有 `templateBatching.promptBudgetChars: 48000`，但本轮按用户确认暂不接入 prompt 字符预算，分批只按模板数量执行。
- 材料分析模板 profile 压缩率约 99.8%，几乎没有压缩；该问题暂缓，后续模板分块时再处理。

## 问题归并

| 编号 | 优先级 | 问题 | 直接影响 | 主要文件 |
| --- | --- | --- | --- | --- |
| P0 | 高 | 运行包版本不明，日志中的 `pwd` 未启用工具直接失败 | 无法判断用户运行的是不是已修复构建 | `apps/desktop/src/main/windowRunService.ts`、`apps/desktop/src/main/taskTextLogger.ts` |

| P2 | 高 | 工具循环多轮的根因没有被定位和消除 | 单窗口可膨胀到 20+ 轮，单纯加上限不能解决根因 | `windowRunService.ts`、`toolErrorClassification.ts`、`taskProgressLog.ts` |
| P3 | 高 | 模板分批规则需要收敛为按模板数量分批 | 避免 prompt 预算分批引入额外复杂度 | `templateBatchPlanner.ts`、`windowRunService.ts`、`defaults.ts` |
| P4 | 高 | full log 写入过重，窗口原文和工具说明占大量篇幅 | 日志大、UI 和人工排查慢 | `windowRunService.ts`、`taskTextLogger.ts`、`taskProgressLog.ts` |
| P5 | 中高 | 模型被迫用 `glob`/`bash` 查报告，但宿主其实知道报告清单 | 每窗口产生可避免工具错误 | `windowRunService.ts`、`globTool.ts`、`toolRegistry.ts` |
| P6 | 中高 | 旧报告应按关键词检索并读取相关段落，而不是整文件读取 | token 膨胀，轮次增加 | `windowRunService.ts`、`grepTool.ts`、`readFileTool.ts` |
| P7 | 中高 | Markdown 写入依赖自由字符串锚点，不稳定 | `old_string` 不唯一/不存在导致反复重试 | `windowRunService.ts`、`multiEditTool.ts`、`editFileTool.ts` |

- [ ] **Step 4: 实现根因汇总，不用上限掩盖问题**

每个窗口维护一份原因计数：

```ts
type ToolLoopRoundReasonSummary = Record<ToolLoopRoundReason, number>;
```

窗口完成或失败时写一条汇总日志。若轮次仍然很多，最终指标要能说明是哪个根因没有消除，而不是只显示“超过上限”。

## 风险与取舍

- 覆盖索引 job 级缓存必须保证保存后内存态同步；否则会出现后续窗口读到旧覆盖状态。
- 日志序列化只能影响写入日志的内容，不能改变真实发给模型的 messages 和 tools。
- 窗口原文只能替换为明确窗口文件名引用，不能把其他 prompt 内容也顺手省略。
- 报告清单只包含任务允许触达的输出文件，不能为了减少查找错误而暴露任意目录。
- 工具循环多轮要靠根因治理减少，不能用硬上限掩盖问题。
- 结构化报告写入接口会扩大工具 schema，需要在旧 `edit_file/multi_edit` 保持兼容的同时逐步迁移。

## 流程沉淀建议

本次问题中有两个流程层面的拖慢点：

1. 运行日志没有 build commit / build time，导致需要人工推断日志是否来自旧构建。
2. 默认 full log 直接铺开窗口原文和完整工具说明，导致普通排查被大体积日志淹没。

建议在项目规则中补充：桌面任务日志必须记录 app version、build commit、build time；完整 message 日志应保留结构和除窗口原文外的内容，窗口原文用当前 `manifestWindow.fileName` 引用，工具说明日志只写工具名和参数。
