# 提取流程批量与模板压缩优化实施文档

## 目标

在不牺牲报告准确性和可审计性的前提下，降低桌面端提取流程的重复输入 token。优先解决同一窗口被多个模板重复发送、模板正文过长、重叠窗口重复提取、模板数量约束与短窗口素材矛盾这四类问题。

本实施文档面向桌面端 TypeScript 流程，Python CLI 只作为已有账本和提示词拆分的参考，不作为本轮主改造目标。

本实施文档及后续拆分计划、验收记录默认使用中文标题、中文文件名和中文正文；除非用户明确要求其他语言，或项目既有规范对特定文件另有要求。

## 目标架构

将当前“窗口内按模板拆成多个 tool-loop”的模式改为“窗口内按可配置批处理策略合并模板”的模式：

1. `TemplateBatchPlanner` 根据配置、prompt 预算、模板风险标签，把同一窗口的模板合并成尽量少的批次。
2. `PromptComposer` 将 prompt 拆成稳定前缀和动态窗口输入：稳定前缀包含系统硬约束和模板压缩卡片，动态部分只包含当前窗口元数据、章节文本和已有报告摘要。稳定前缀优先放入 system message 或等价缓存友好的消息位置。
3. `TemplatePromptProfile` 从完整模板生成面向 LLM 的压缩卡片；完整模板继续保存在规则快照和 hash 中，保证审计能力不丢。
4. `BatchOutcomeTracker` 按唯一模板输出键记录每个模板在本批次内的结果，避免合并后某个模板被模型漏处理。若本轮采用 `outputFileName` 作为 outcome 键，任务创建或运行开始前必须先拒绝重复 `outputFileName`；否则 tracker 必须使用 `templateId + outputFileName` 复合键。
5. `ReportCoverageIndex` 按书籍、模板、输出文件、提交章节范围、窗口 hash、模板 hash 记录覆盖状态，在模型调用前决定跳过、缩小批次或补提。覆盖索引不能放在模型可通过 `read_file/grep reports` 读取的位置；如确需靠近 reports 目录，必须由 `ToolReadPolicy`/`ReportPathPolicy` 明确排除。
6. `ReportPathPolicy` 保持当前扁平 reports 输出为默认策略；路径分层作为后续可选能力，不在第一阶段强改。
7. `ToolReadPolicy` 为旧报告读取、去重诊断和错误恢复提供分页/片段读取能力，避免把整份旧报告回灌给模型。

## 非目标

- 不在本轮引入 fine-tune。
- 不把特定小说、特定 4 个模板、特定章节号写死到代码中。
- 不删除完整模板快照；压缩只影响 LLM prompt 输入。
- 不立即重构整个桌面运行时，不把所有逻辑继续堆进 `windowRunService.ts`。
- 不把 reports 默认输出改成子目录，避免破坏现有报告列表、预览、注册和安全边界。

### 任务合同

本分支目标不是一次性把所有设想塞进运行时，而是按可验证的阶段收敛：先封住多模板合并的正确性风险，再开启默认合并，然后做模板 prompt 压缩，最后接入覆盖索引。阶段 5 到阶段 8 暂作为后续增强阶段保留清晰接口和测试入口；阶段 9 的真实 UI 与日志验收是最终完成声明前的强制闸门，不属于可延后的增强项。若阶段 0.5、阶段 3、阶段 1、阶段 2、阶段 4 没有稳定通过，不进入后续阶段编码。

本分支计划和编码前的高风险决策必须保留多 agent 审查闭环：复杂阶段计划、默认多模板合并前、覆盖索引/路径策略/内容安全策略变更前，应按架构边界、配置化覆盖、边界与安全、验收可观测性分派 reviewer agent；主 agent 汇总审查发现，先修订实施文档并消除 required finding，再进入编码或继续下一阶段。

每个阶段必须满足四个闸门：

1. **RED 闸门**：先写一个会失败的目标测试，失败原因必须是当前缺失行为，而不是测试拼写或 fixture 错误。
2. **GREEN 闸门**：只实现让当前测试通过的最小代码，并运行该测试。
3. **回归闸门**：运行本阶段相关包测试、`pnpm typecheck` 的必要子集；阶段 0.5、阶段 3、阶段 1、阶段 2、阶段 4 完成后运行 `pnpm test:ts` 和 `pnpm typecheck`。
4. **桌面验收闸门**：不能只靠脚本测试宣布完成；阶段 0.5、阶段 3、阶段 1、阶段 2、阶段 4 集成后必须打开桌面 UI，用真实小说运行，并检查任务日志文件。

## 阶段 1：同窗口多模板合并

## 阶段 3：批次结果跟踪，防止合并后漏模板

### 设计要点

合并请求后，一个模型调用要负责多个输出文件。只要求最终文本 `NO_UPDATE` 无法表达“甲模板写了、乙模板无更新、丙模板漏处理”的差异，因此需要按输出文件记录 outcome。

### 实施步骤

1. 新增 `BatchOutcomeTracker`，初始化时登记本批次所有 `outputFileName`。
2. 写工具成功后，将对应输出文件标记为 `written`。
3. 增加轻量的无更新表达机制。建议新增一个只记录状态、不写文件的工具，例如 `mark_no_update`，参数为 `path` 和原因；或者定义可机器解析的最终文本格式。优先选择工具方式，因为它能复用现有 tool-loop 校验。
4. 当模型没有给所有输出文件提供 outcome 时，不直接完成本批次，而是追加纠正消息，要求补齐缺失输出文件的写入或无更新标记。
5. 当所有输出文件都有 outcome 后，才允许本批次完成。
6. 记录 outcome 到日志和覆盖索引，区分 `written`、`no_update`、`skipped_covered`、`failed`。

### 验收标准

- 同一批次 4 个模板必须最终产生 4 个 outcome。
- 只写入 1 个报告但遗漏其他 3 个模板时，tool-loop 会进入纠正轮，而不是直接完成窗口。
- `NO_UPDATE` 仍可用于“本批所有模板都无更新”的快速完成路径。

## 阶段 4：增量去重与覆盖索引

### 设计要点

不建议把去重建立在模型写入前临时 `grep` 章节号上。报告文本格式会变化，章节号写法也会变化。更稳的做法是维护机器可读的覆盖索引，`grep` 只作为旧报告迁移或人工排查的辅助手段。

### 实施步骤

1. 新增 `ReportCoverageIndex`，默认位置应为模型工具不可读的书籍元数据目录，例如 `assets/books/<bookId>/.coverage/coverage.json`，或由 `coverageIndexDefaults.pathTemplate` 配置；不得默认放在 `reports/.index/coverage.json`，除非同阶段工具策略已证明模型无法读取该目录。
2. 覆盖记录至少包含：`bookId`、`jobId`、`windowId`、`outputFileName`、`templateId`、`templateHash`、`templateGroupHash`、`outputFileNamesHash`、`windowHash`、`submittedChapterRange`、`submittedChapterTitles`、`outcome`、`updatedAt`、`rulesSemanticHash`。
3. 在运行窗口开始前，如果 `skipAlreadyExtracted` 为 true，先用覆盖索引判断当前窗口提交章节和模板是否已经覆盖。
4. 如果 `skipAlreadyExtracted` 为 false，不做模型调用前跳过，但仍更新覆盖索引，保证后续任务可复用。
5. 若本批所有模板都已覆盖，跳过模型调用并记录 `skipped_covered`。
6. 若只有部分模板已覆盖，从当前批次中剔除已覆盖模板，只把未覆盖模板送入 `TemplateBatchPlanner`。
7. 写工具成功或 `mark_no_update` 成功后更新覆盖索引。
8. 对没有索引的旧报告，可提供一次性迁移或保守策略：不自动判定已覆盖，除非报告中存在可机器识别的来源标记；人工 grep 结果只作为提示，不直接跳过。
9. 将覆盖索引写入做成原子更新，避免任务失败时留下半写状态。
10. 引入 `read_file` 分页或片段读取策略，避免为判断旧内容而把长报告整篇注入下一轮 messages。

### 验收标准

- 第二次运行同一书籍、同一模板、同一窗口 hash 时，模型请求数为 0 或只处理未覆盖模板。
- 重叠窗口只默认处理 `submittedChapterRange`，前置重叠章节不触发重复提交。
- 修改模板正文、规则语义、窗口文本任一 hash 后，对应覆盖记录失效，需要重新处理。
- 覆盖索引损坏时任务给出明确错误或保守重跑，不静默跳过。
- 模型不能通过 `read_file` 或 `grep reports` 读取覆盖索引内容。

## 阶段 5：章节边界结构化

### 设计要点

当前窗口文本以 Markdown 标题拼接，模型需要自行识别章节边界。可以在 prompt 层引入 XML 风格章节标签，但不一定要改变磁盘上的 `window-0001.txt` 文件。

### 实施步骤

1. 在 `RuntimeWindowManifestWindow` 中补充 `contextChapterIds` 与 `submittedChapterIds`，让 prompt formatter 不依赖标题文本倒推章节号。
2. 新增 `WindowPromptTextFormatter`，输入 manifest window 和窗口文本，输出带章节标签的 prompt 文本。
3. 标签建议包含 `id`、`title`、`submitted` 三类信息，例如章节是否属于本轮提交范围。
4. 保留原始窗口文件格式，降低对现有 read_file/grep 测试和人工排查的影响。
5. 修正 `toJobWindowInputs()` 的 `chapterIds` 来源，避免继续把标题当章节 ID 传入 job runtime。
6. 修改 `buildWindowPrompt` 的“当前窗口文本”部分，使用 formatter 生成的结构化文本。
7. 更新测试，断言 prompt 中存在章节标签、提交章节标记和原文正文，并且不丢失中文标题。

### 验收标准

- 模型请求中每个章节都有明确边界。
- 重叠章节和提交章节可由标签区分。
- 章节标签不进入正式报告；写前 guard 继续拦截内部窗口文件名和运行路径。

## 阶段 6：约束分层与后处理

### 设计要点

当前 `toolLoopDefaults.windowInstructionLines` 同时承载硬约束、质量要求和可由程序检查的格式要求。需要拆层，减少 prompt 负担，同时保持安全边界。

### 实施步骤

1. 在配置中拆分规则层级：`p0HardRules`、`qualityRules`、`formatRules`、`postWriteGuards`。
2. P0 硬约束继续进入每次请求，例如：只能基于当前窗口、写工具 path 必须来自 `outputFileName`、不能使用内部路径。
3. P1 质量规则压缩为短句进入 prompt，例如：未说明写“原文未说明”、不得虚构、来源必须可追溯。
4. P2 格式规则优先转为程序 guard 或 post-processor，例如模板状态、内部路径、窗口文件名、标题去“模板”。
5. 将现有内部元数据拦截逻辑从 `windowRunService.ts` 中抽到 `ReportContentPolicy`，避免主流程继续膨胀。
6. 为每类规则配置独立测试：prompt 长度测试、guard 拦截测试、post-processor 规范化测试。

### 验收标准

- prompt 中重复规则数量下降。
- 内部路径、`window-0001`、`状态：模板/草案` 等仍会被写前拦截。
- 新增规则通过配置增加，不需要在 `windowRunService.ts` 中散落新正则。

## 阶段 7：事件因果链数量策略配置化

### 设计要点

“每部作品至少 10 条长程因果链”属于全书级目标，不适合强制每个短窗口完成。该规则应从模板正文里的自然语言硬约束沉淀为模板策略。

### 实施步骤

1. 新增模板级 `quantityPolicy` 配置，至少支持：全书目标数、单窗口最小数、素材不足时是否允许 0 条、是否要求跨章节/跨地点/跨势力。
2. 对“事件因果链（长程因果图）”配置为：全书目标可为 10；单窗口最小为 0；只有满足长程特征时才输出；素材不足时使用 `mark_no_update` 或报告“原文未说明”。
3. 将数量策略写入模板 prompt profile，而不是依赖完整模板正文中的自然语言。
4. 增加测试：短窗口只有单场景规则突破时，不强迫模型输出长程因果链。
5. 增加测试：跨章节、跨地点、跨势力素材充足时，仍会输出长程因果链。

### 验收标准

- 早期短窗口不会因为全书目标被迫硬凑条目。
- 全书完成后可用覆盖索引或报告统计检查总目标达成情况。
- 数量策略可配置，不写死在某个模板名称上。

## 阶段 8：报告路径策略

### 设计要点

当前架构已经按书籍隔离到 `assets/books/<bookId>/reports`，真正缺的是模板类别组织。考虑现有报告列表和安全校验，第一阶段保持扁平目录更稳。

### 实施步骤

1. 保持 `write_file/edit_file/multi_edit` 默认只接受裸 `outputFileName`。
2. 在模板保存和任务开始前校验同一书籍内 `outputFileName` 唯一。
3. 抽出 `ReportPathPolicy`，集中处理 reports root、项目相对 reports root、`reports/<file>` 别名、裸报告名、当前书籍限制、本批 output 白名单。
4. 底层 `ReportWriter` 继续只负责真实路径安全和写入语义，不理解小说窗口和模板批次。
5. 如需分类，优先通过 `outputFileName` 前缀或模板元数据展示分类，不立即允许模型写子目录。
6. 若后续必须支持子目录，扩展 `ReportPathPolicy`：`flat` 为默认，`categorySubdir` 为显式配置；同步更新 `reportWriter`、资产注册、列表预览、路径安全测试。

### 验收标准

- 本轮优化不破坏现有报告预览和注册。
- 模型仍不能随意写目录路径。
- 同名输出文件在任务创建或运行前被明确拒绝。

## 主要风险与对策

| 风险 | 对策 |
| --- | --- |
| 合并后模型漏处理某个模板。 | 阶段 3 的 `BatchOutcomeTracker` 要求每个输出文件都有 outcome。 |
| 两个模板使用同一输出文件导致 outcome 被折叠。 | 阶段 0.5 在任务创建或运行前拒绝重复 `outputFileName`，或明确改用复合 outcome 键。 |

| 覆盖索引被模型通过 reports 目录读取。 | 默认放在模型工具不可读目录，或由 `ToolReadPolicy`/`ReportPathPolicy` 排除索引目录并测试。 |

| 改 reports 子目录影响 UI。 | 第一阶段保持扁平目录，只做 outputFileName 唯一性和可选后续策略。 |
| 旧测试大量依赖单模板批次。 | 先重写语义测试：从“请求隔离”改为“结果完整、可配置拆批、预算触发拆批”。 |
