# 模型自动切换与失败续跑 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让提取任务在 API 限流、额度不足、超时等可恢复异常下自动续跑，并把模型选择从长列表改成“自动/供应商大类 + 子模型”的清晰选择。

**Architecture:** 供应商配置仍以 `ProviderConfig` 为源头，利用现有 `models[].isDefault` 表示每个已保存大类的默认模型；模型选择 UI 从扁平模型列表升级为 provider-first 视图。任务输入持久化 `modelSelectionMode` 和 `autoRetryOnFailure`，主进程在执行期根据已保存配置显示顺序生成循环 fallback 计划，按 `1 -> 2 -> 3 -> 1 -> 2 -> 3` 继续尝试。失败续跑由桌面端主进程的轻量 retry scheduler 负责，5 分钟间隔和可切换错误规则进入项目配置，避免硬编码到组件或 handler 里。

**Tech Stack:** Electron main process、React、TypeScript、Vitest、现有 IPC `jobs:*`/`providers:*`、`@novel-extractor/config`、`@novel-extractor/llm`、`windowRunService`、`jobScheduler`。

## 需求边界

- 任务卡片增加“失败后自动续跑”勾选项，默认关闭。
- 勾选后，任务进入 `failed` 时每 5 分钟尝试继续执行一次。
- 一旦自动尝试成功提交给调度器，状态进入 `running` 或 `queued`，本轮 5 分钟定时器取消，不再重复触发。
- 如果自动尝试仍失败且任务仍处于 `failed` 且勾选未取消，则继续等待下一个 5 分钟周期。
- 模型设置改为两级选择：先选“自动”或已配置且启用的大类，再选该大类下启用的子模型。
- “自动”模式按已保存配置列表的显示顺序使用每个大类的默认模型，遇到额度不足、限流、超时等可切换错误时切到下一个大类默认模型。
- 自动模式循环尝试，不把已失败过的大类永久排除。示例顺序为 `1 -> 2 -> 3 -> 1 -> 2 -> 3`。
- 为避免瞬时无限循环，单个窗口内的立即 fallback 轮数走配置；超过阈值后任务先失败，若勾选了自动续跑，则 5 分钟后继续按循环游标尝试。
- 已保存配置列表每项增加编辑按钮，可修改默认模型、启用/停用子模型、补充或修改子模型型号。

## 非目标

- 不新增供应商优先级配置；自动顺序直接使用已保存配置显示顺序。
- 不实现供应商级 RPM/TPM 精细限速器。
- 不让工具参数错误、报告写入安全拦截、路径越权等非模型问题触发模型切换。
- 不把失败供应商写入黑名单，也不做“本轮试过就不再试”的去重策略。
- 不自动提交代码；执行时如需提交，必须等用户明确授权。

## 数据结构设计

### 任务输入

`CreateJobDto` 保持旧字段可用，新增模式字段。旧任务没有这些字段时按 explicit 处理。

```ts
export type JobModelSelectionMode = "explicit" | "auto";

export interface CreateJobDto {
  bookId: string;
  templateIds: string[];
  providerConfigId: string;
  modelId: string;
  modelSelectionMode?: JobModelSelectionMode;
  autoRetryOnFailure?: boolean;
  singleRunChapterCount: number;
  extractionChapterCount: number;
  overlapChapterCount: number;
  skipAlreadyExtracted: boolean;
}
```

语义：

- `modelSelectionMode` 缺省为 `"explicit"`。
- explicit 模式下 `providerConfigId/modelId` 就是用户明确选择的供应商和子模型。
- auto 模式下 `providerConfigId/modelId` 保存创建时解析到的第一个候选模型，便于旧日志和列表显示；执行期仍以 providerStore 当前已保存配置顺序生成 fallback 计划。
- `autoRetryOnFailure` 缺省为 `false`。

### 任务展示

`InputSummaryDto` 增加可选字段，避免把自动模式只显示成某个模型名。

```ts
export interface InputSummaryDto {
  bookDisplayName: string;
  templateNames: string[];
  modelId: string;
  modelSelectionMode?: JobModelSelectionMode;
  currentProviderDisplayName?: string;
  currentModelDisplayName?: string;
}
```

展示建议：

- explicit：`模型：MiniMax / MiniMax-M3`
- auto：`模型：自动 - 当前 MiniMax / MiniMax-M3`
- 自动切换时通过 `jobs:updated` 更新当前 provider/model 展示。

## 执行语义

### 自动模型循环

候选集只来自已保存配置列表中满足条件的 provider：

- provider 已启用。
- provider 有 API key。
- 至少一个模型已启用。
- 至少一个已启用模型是默认模型；如果历史数据没有默认模型，则使用第一个启用模型并在日志中记录兼容 fallback。

循环规则：

```text
providers: [MiniMax, DeepSeek, Claude]
defaults:  [MiniMax-M3, DeepSeek V4 Flash, claude-fable-5]
sequence:  MiniMax-M3 -> DeepSeek V4 Flash -> claude-fable-5 -> MiniMax-M3 -> ...
```

自动切换不修改 provider 配置，不把失败项从候选集中删除。

### 失败续跑

`failureRetryScheduler` 只管理任务级 timer，不直接执行窗口逻辑。

```text
job.failed + autoRetryOnFailure=true
  -> schedule now + failureRetryIntervalMs
  -> timer fires
  -> enqueueModelBackedJob(job, { skipAlreadyExtracted: true })
  -> enqueue 成功 running/queued: cancel timer
  -> enqueue 抛错或任务仍 failed: schedule next interval
```

取消条件：

- 用户取消勾选。
- 用户删除任务。
- 用户手动开始、继续或重新开始成功提交给调度器。
- 任务进入 completed。
- 应用关闭时自然清理内存 timer；下次加载项目 runtime 后，已失败且勾选的任务重新安排一个完整 5 分钟周期。
