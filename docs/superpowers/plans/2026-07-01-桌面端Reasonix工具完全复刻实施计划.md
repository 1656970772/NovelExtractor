# Desktop Reasonix Tools Full Port Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将 Reasonix 的 `read_file`、`write_file`、`edit_file`、`multi_edit`、`grep`、`glob`、`ls`、`bash` 八个工具完整移植到 NovelExtractor 桌面端工具循环。

**Architecture:** 在桌面端工具层建立 Reasonix 风格工具实现与注册入口，由桌面运行时统一使用该工具集。每个工具作为独立任务实现和验收，共享能力沉淀到独立基础模块，避免把工具细节散落在 `windowRunService` 或 handler 中。

**Tech Stack:** TypeScript、Node.js、Vitest、Electron main process、现有 `@novel-extractor/tools` / `@novel-extractor/config` / `apps/desktop`。

---

> **切记：本计划要求完全复刻 Reasonix 项目的工具代码行为，除了程序语言改为 TypeScript，其他模型可见协议、运行逻辑、返回内容、错误语义和边界行为都要一样。执行过程中不要出现“暂时不做”“下个阶段做”“后续再补”这类逻辑。**

## 范围

- 只面向桌面版工具循环；CLI 后面会删除，本计划不为 CLI 做兼容设计。
- 工具集合固定为：`read_file`、`write_file`、`edit_file`、`multi_edit`、`grep`、`glob`、`ls`、`bash`。
- 会话待办清单与完成签收工具不属于本次工具集合。
- 以 `E:\Github_Projects\DeepSeek-Reasonix` 中对应 Go 源码为行为基准。
- 允许 TypeScript 实现根据语言和运行时做自然表达，但最终外部行为必须与 Reasonix 对齐。
- 改造这些工具后，相关调用方也可以同步调整；优先让项目代码兼容 Reasonix 工具协议和行为，而不是让工具迁就旧调用方式。

## 推荐文件边界

- `packages/tools/src/reasonix/`：Reasonix 风格工具主目录。
- `packages/tools/src/reasonix/tools/`：八个工具分别落文件。
- `packages/tools/src/reasonix/workspace.ts`：桌面端绑定工作目录、写入根、只读根、搜索配置和 shell 配置的统一入口。
- `packages/tools/src/reasonix/registry.ts`：统一导出工具定义、工具执行和工具顺序。
- `packages/tools/src/reasonix/pathResolver.ts`：路径解析、read alias、写根限制和禁止读取根。
- `packages/tools/src/reasonix/encoding.ts`：文件读写编码处理。
- `packages/tools/src/reasonix/textEdit.ts`：文本替换、换行处理、唯一性判断。
- `packages/tools/src/reasonix/searchWalk.ts`：搜索类工具的通用遍历能力。
- `packages/tools/src/reasonix/shell.ts`：shell 解析、执行、超时和进程处理。
- `packages/config/src/defaults.ts`：桌面端默认启用工具配置。
- `apps/desktop/src/main/windowRunService.ts`：接入统一 Reasonix 工具 registry，不承载具体工具实现。

这些文件边界是建议结构；执行者可以在不破坏职责边界的前提下调整命名和拆分。

## 实施策略

- 每个 Task 开始前先读对应 Reasonix 源文件和测试文件，确认行为边界。
- 每个 Task 先补 parity 测试，再实现，再跑目标测试。
- 每个工具完成后独立提交，方便回滚和审查。
- 测试重点由 Reasonix 对应源码和测试决定，执行者按 parity 需要自行展开。
- 不把 Reasonix 源码逐行翻译成僵硬结构；允许用 TypeScript 写得自然，但不能改变行为。
- 不把领域规则、小说规则、模板规则写进工具层。工具层只做通用 Reasonix 工具能力。
