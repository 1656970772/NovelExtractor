## 当前问题

### 问题：集成测试超时

**症状：**
- 测试 "starts jobs for different books concurrently" 超时（5秒）
- 测试 "queues a second job for the same book while another book can run" 超时（5秒）

**根本原因分析：**
1. 调度器的 `run` 函数异步调用 `runModelBackedJob`
2. `enqueueModelBackedJob` 在调度器启动任务后立即返回
3. `runModelBackedJob` 内部才更新状态为 `running`，但这发生在 `enqueue` 返回之后
4. 测试期望 `jobs:start` 返回 `status: "running"`，但实际返回 `status: "created"`
5. 尝试轮询等待状态更新导致测试超时

**已尝试的解决方案（均失败）：**
1. 在 `onStarted` 回调中更新状态 → 导致死锁/超时
2. 在 `enqueueModelBackedJob` 中轮询等待状态更新 → 超时
3. 使用 `setImmediate` / `setTimeout` 等待 → 状态仍然是 `created` 或超时

## 下一步行动

### 方案 A：调整测试期望（推荐）
- 接受 `jobs:start` 立即返回时状态可能是 `created`
- 修改测试使用 `vi.waitFor` 等待状态变为 `running`
- 这更符合异步调度的语义

### 方案 B：同步状态更新
- 在调度器的 `startEntry` 中，在 `addRunning` 之后同步调用状态更新
- 需要确保不会导致死锁
- 可能需要重构 `updateJob` 以支持同步调用

### 方案 C：使用事件机制
- 在 `runModelBackedJob` 开始时发出事件
- `enqueueModelBackedJob` 等待该事件
- 需要添加 EventEmitter 或类似机制
