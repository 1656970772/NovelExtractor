# 桌面端完整提取流程设计

## 背景

当前桌面端点击“开始”后的提取流程没有复刻 CLI 的实际流程。现状更接近“一次性把选中模板和章节拼成 prompt，请求一次模型，再把模型返回包装成一个报告”，这会带来三个核心问题：

- 结果格式错误：命中多个模板时没有生成多个对应 Markdown，而是生成一个被桌面端包装过的报告。
- 运行状态不实时：`jobs:start` 等模型请求和写入全部结束后才返回，UI 只能显示开始和完成，无法在每个运行窗口、每次模型调用后刷新。
- usage 统计不完整：只显示总 Token 和费用，缺少输入、输出、缓存命中，且费用应从 UI 移除。

本设计以 CLI 的提取语义为基准，桌面端必须完整复刻 CLI 的窗口切分、模板路由、工具写入、进度账本、usage 统计与报告产物行为。同时针对 CLI 已暴露的问题做桌面化改进：结构化事件流、结构化 usage、明确 UI 刷新策略、报告产物注册、可恢复错误和可审计日志。

## 目标

用户点击任务卡上的“开始”后，桌面端应执行完整提取流程：

1. 读取任务快照：书籍原文资产、运行窗口 manifest、模板快照、模型配置、输出目录。
2. 按本次任务生成的运行窗口文件逐个运行。
3. 每个窗口内先按 `skipAlreadyExtracted` 和进度账本判断可跳过的窗口-模板组；已命中的账本单元跳过该组全部模型调用，包括路由调用。剩余候选组无可用路由缓存或账本命中时，才执行大模型模板路由。
4. 每次任务运行前，先根据原文和窗口参数生成本次运行的 `window-xxxx.txt` 文件，再根据本次选择的模板生成运行级不可变规则快照 `runs/<jobId>/rules/提取规则.md`，并同步项目级 `rules/提取规则.md` 作为 latest copy / 人工入口。
5. 每次模型调用通过 OpenAI-compatible tool calling 循环执行。
6. 模型写入前先查询已有 Markdown，重复完整则跳过，不完整则补充或修改。
7. 命中几个输出模板，就生成或更新几个 Markdown 文件。
8. 用户可通过任务参数 `跳过已提取` 决定是否跳过账本中已完成且未变化的窗口模板组。
9. UI 在任务开始、规则文档生成、窗口开始、模板组开始、模型调用开始、每次 usage 返回、工具调用、文件写入、窗口完成、任务完成时刷新。
10. 任务卡显示输入 Token、输出 Token、缓存命中 Token，不显示费用。
11. 生成的 Markdown 自动进入书籍资产页的报告列表，可直接预览。

## 非目标

- 不做“最小可接受实现”或临时单次模型请求路径。
- 不继续把模型最终文本包装成桌面端自定义报告。
- 不把费用展示在任务卡、日志摘要或结果区域。
- 不在本设计中实现图谱导入、世界观卡片生成、模板市场或复杂报告版本管理。
- 不自动删除共享 Markdown 报告。删除任务默认只删除任务记录和运行日志索引。

## 术语

- 运行窗口：一次给模型看的原文片段，由“单次运行章节数”“重叠章节数”和“提取章节数”规划，并在任务运行目录中落为 `window-xxxx.txt`。重叠章节只做上下文，不作为本轮新增提交。
- 章节边界：切分脚本识别出来的临时结构，只用于生成运行窗口和日志范围，不再作为长期章节 txt 资产保存。
- 模板快照：创建任务时保存的模板名称、模板正文、输出文档名、路由说明、分组信息、hash 和创建时间。任务运行使用快照，不受后续模板编辑影响。
- 提取规则文档：每次运行生成一个不可变规则快照 `runs/<jobId>/rules/提取规则.md`，项目级 `rules/提取规则.md` 只是 latest copy / 人工入口。规则文档记录模板分组、路由说明、输出文件名、通用抽取约束和写入约束，是大模型路由的输入依据，不是 Runtime 用来做关键词硬匹配的代码规则。
- 输出文档：模板对应的 Markdown 文件，例如 `丹药分析.md`。
- 模板命中：大模型路由根据当前运行窗口内容和 `提取规则.md` 返回的结构化命中结果。
- usage：模型返回的输入、输出、总量、缓存命中、缓存未命中等 token 统计。
- 工具循环：模型调用 `read_file`、`grep`、`write_file`、`edit_file`、`multi_edit` 等工具，直到最终返回 `NO_UPDATE` 或完成。

## 已确认讨论结论

### 运行窗口

桌面端保留用户可理解的三个参数：

- `singleRunChapterCount`：每个窗口最多看几章。
- `overlapChapterCount`：下一个窗口向前重叠几章作为上下文。
- `extractionChapterCount`：本次最多处理多少章。

上传小说时只保存原文资产，不再生成 `assets/books/<bookId>/chapters/*.txt` 这类长期章节文件。每次任务开始执行时，Runtime 读取原文，调用切分脚本生成本次任务专用的运行窗口文件。

窗口规划规则：

- `overlapChapterCount >= 0`。
- `overlapChapterCount < singleRunChapterCount`。
- `stride = singleRunChapterCount - overlapChapterCount`。
- 第一个窗口提交窗口内全部章节；后续窗口只提交非重叠章节。
- 重叠章节只帮助模型理解上下文，不重复写入账本，也不重复当成本轮新增内容提取。

示例：

```text
singleRunChapterCount = 5
overlapChapterCount = 1
extractionChapterCount = 13

窗口 1：上下文 第1-5章，提交 第1-5章
窗口 2：上下文 第5-9章，提交 第6-9章
窗口 3：上下文 第9-13章，提交 第10-13章
```

日志必须显示上下文范围和本轮提交范围，例如：

```text
窗口 2/10：上下文 第5-9章，本轮提交 第6-9章
```

窗口文件命名固定为：

```text
runs/<jobId>/windows/window-0001.txt
runs/<jobId>/windows/window-0002.txt
runs/<jobId>/windows/manifest.json
```

### 模板路由

模板是否命中不由代码按关键词直接判断。正确流程是：

```text
窗口开始
  -> 对候选窗口-模板组查询进度账本和路由缓存
  -> 已命中账本的模板组跳过全部模型调用
  -> 对剩余候选组读取本次任务规则快照
  -> 把当前窗口文本、模板快照摘要、模板组摘要、提取规则交给路由模型
  -> 路由模型返回结构化 JSON
  -> Runtime 校验 groupId/templateId/outputFileName 是否属于本次快照
  -> 只对命中的模板组进入正式提取工具循环
```

`提取规则.md` 的作用是把模板说明、模板分组、输出文件和路由注意事项讲清楚，让模型判断“当前窗口是否应该跑这个模板”。它不是代码的确定性关键词表。

路由调用本身是一次大模型调用，必须进入 usage 和日志：

- 计入输入、输出、缓存命中、缓存未命中。
- 日志显示窗口、上下文范围、本轮提交范围、候选模板组、命中模板、跳过模板、路由原因。
- 路由模型不能写文件，只能返回结构化路由结果。

路由结果必须结构化，但 JSON 必须尽量简单。模型只返回 Runtime 真正需要的字段：

```json
{
  "matchedGroupIds": ["resource_group", "faction_group"],
  "skipped": [
    {
      "id": "npc_group",
      "reason": "本窗口没有角色性格变化"
    }
  ]
}
```

Runtime 根据 `matchedGroupIds` 从模板组快照展开 `templateIds`、`outputFileNames` 和中文显示名。路由模型不返回模板 ID、输出文件名或 confidence，减少非法字段和解释空间。用户可见日志显示中文名，例如“神兵利刃分析”，不显示内部 ID。

若路由模型不确定，应保守命中，而不是跳过；如果 `skipped.reason` 出现“不确定”“可能相关”等表达，Runtime 要求重试或按保守命中处理。这条规则用于避免漏提取。

### 模板快照

创建任务时必须保存模板快照和模板组快照。任务开始、暂停、继续、重跑都使用快照，不受用户后续编辑模板影响。

模板快照至少包含：

```text
templateId
templateName
templateBody
outputFileName
routeCard / routeDescription
groupId
templateHash
createdAt
```

模板组快照至少包含：

```text
groupId
templateIds
maxFullTemplatesPerCall
groupHash
```

后续判断“账本是否可跳过”时使用窗口 hash、模板 hash、模板组 hash、输出文件名和 `rulesSemanticHash`。模板内容变了，新任务使用新快照；旧任务继续按旧快照跑。

### 旧报告查询

Runtime 不预先帮模型从目标 Markdown 中找相关片段塞进 prompt。正式提取时，模型必须自己调用工具查询目标报告：

```text
read_file / grep -> 判断已有内容 -> write_file / edit_file / multi_edit 或 NO_UPDATE
```

Runtime 要校验：既有报告在第一次修改前必须出现过针对该文件的 `read_file` 或 `grep`；新报告可以直接 `write_file` 创建；同一 tool loop 内已成功创建的同一报告，可在后续轮次继续用 `edit_file` / `multi_edit` 修正，不要求重新查询。跨窗口、新 tool loop 或运行前已存在的旧报告仍必须先查询。

### 工具、预算和日志

工具系统参考 Reasonix：

- 工具用稳定 schema 暴露给模型。
- 配置中 `enabledTools = []` 表示默认全工具。
- 任务开始后冻结工具快照和工具预算。
- 只读工具可以并行，写工具和未知工具串行。
- 工具结果超长保留头尾并记录截断事件。

Prompt budget 和工具输出预算都走配置或任务快照：

- 路由调用和正式提取调用都要统计 prompt budget。
- 工具输出先按预算截断，再进入下一轮 prompt。
- 达到 warn 阈值只记日志；达到 hard 阈值当前窗口失败，不静默裁剪窗口正文。

日志按事件从上到下追加，第一条在最上面。日志要尽可能接近 Codex 运行轨迹：显示每次大模型请求、响应、usage、工具 dispatch、工具结果、文件写入和错误，但不显示外部模型隐藏推理内容。

左侧导航栏底部提供“设置”入口，打开和模板选择弹窗同规格的设置弹窗。当前只包含“日志与隐私”分类，分类内只有一个设置项：`调试日志`。`调试日志` 默认开启，用于记录结构化调试事件和可审计摘要。

## 总体结论

桌面端不应在 `apps/desktop/src/main/p0Handlers.ts` 内直接拼 prompt、请求一次模型、写一个报告。应建立桌面端提取运行时，复用并对齐 CLI 的核心流程：

```text
点击开始
  -> 校验任务和模型配置
  -> 创建运行快照和事件订阅
  -> 读取原文并生成/刷新本次运行窗口文件
  -> 生成运行级规则快照并同步项目级 latest 提取规则.md
  -> 读取窗口 manifest 并确认执行顺序
  -> 对每个窗口先做账本/路由缓存跳过检查
  -> 对剩余候选模板组执行大模型模板路由
  -> 对每个命中模板组执行模型工具循环
  -> 工具读旧报告并写 Markdown
  -> usage 和日志实时入账
  -> 注册写入报告资产
  -> 任务完成或失败
```

## 模块边界

### Renderer

Renderer 只负责展示和用户操作：

- 点击“开始”“暂停”“继续”“删除”“查看结果”。
- 订阅或轮询任务事件。
- 将任务事件投影为任务卡状态、进度、usage、日志和写入文件列表。
- 点击“查看结果”时跳转资产页，打开该任务生成或更新的 Markdown 报告列表。

Renderer 不负责拼 prompt、解析章节、计算模板命中、处理工具调用或写文件。

### Preload IPC

Preload 暴露类型安全 API：

- `jobs:start`：启动任务，立即返回启动后的任务状态，不等待任务完成。
- `jobs:get` 或 `jobs:list`：读取任务当前快照。
- `jobs:events`：订阅任务事件。若 Electron IPC 订阅暂时不方便，可用短轮询作为过渡，但运行时内部仍以事件为真实状态来源。
- `books:listReports`：读取书籍下当前报告资产。
- `reports:preview`：预览 Markdown。

推荐使用事件订阅，短轮询只作为兼容 fallback。

### Main Job Orchestrator

Main 进程负责任务编排：

- 校验任务是否存在、是否允许开始。
- 加载书籍原文资产、窗口 manifest、模板快照、模型配置。
- 按任务参数生成或刷新本次运行窗口文件。
- 根据任务选择的模板生成运行级规则快照，并同步项目级 latest `提取规则.md`。
- 创建运行目录上下文。
- 启动后台任务，不阻塞 `jobs:start` IPC。
- 维护单任务运行限制。
- 转发运行时事件到 Renderer。
- 持久化任务状态、事件、usage、写入文件和失败原因。

### Extraction Runtime

Extraction Runtime 是桌面端完整提取流程的核心：

- 生成或读取运行窗口文件。
- 构造窗口上下文和提交范围。
- 读取本次任务的规则快照路径和项目级 latest 路径。
- 计算窗口 hash、模板 hash、输出 hash 和 `rulesSemanticHash`。
- 查询进度账本，已完成且 hash 未变化的窗口模板组跳过；跳过命中时不再为该窗口-模板组发起路由或正式提取模型调用。
- 对剩余候选模板组读取可用路由缓存；无缓存或缓存 hash 不匹配时再路由命中模板或模板组。
- 构造 system prompt 和 user prompt。
- 调用 LLM tool loop。
- 校验工具写入结果。
- 写入或更新进度账本。
- 发布结构化事件。

### LLM Adapter

LLM Adapter 对齐 CLI 的 OpenAI-compatible tool calling 能力：

- 支持 `messages + tools + tool_choice=auto`。
- 工具以协议形式暴露给模型：`name`、`description`、JSON Schema、`readOnly`、执行器。
- 路由模型调用不发送工具列表，也不允许工具调用；路由只返回结构化 JSON。
- 正式提取模型调用才发送工具列表；每一轮 tool loop 都带同一份任务快照中的稳定工具 schema。
- 每次请求发送稳定排序后的工具 schema；schema 内容需要稳定序列化，避免同一工具列表因顺序变化破坏缓存。
- 处理多轮 tool calls。
- 支持流式 tool call start：当模型刚返回工具名时即可记录日志，完整参数流完后再进入工具执行。
- 每一轮模型响应后立即 normalize usage。
- 通过事件上报本轮 usage，并累加到任务 usage。
- 支持 DeepSeek 顶层缓存字段和 OpenAI `prompt_tokens_details.cached_tokens`。
- 工具错误作为 tool result 返回给模型，除非属于认证、写入边界、任务取消等不可恢复错误。

### Tool Executor

工具执行器绑定当前项目和当前书籍 reports 目录：

- 参考 Reasonix 的工具注册和执行模型，不把工具调用做成 prompt 文本约定。
- 默认工具配置为全工具：`enabledTools = []` 表示启用内置工具全集。
- 创建任务或开始任务时冻结工具快照：`toolProfile`、`enabledTools`、`toolSchemaHash`、工具预算配置。冻结时只做运行准备，不发送给模型。
- 读工具：`ls`、`read_file`、`grep`、`glob`。
- 读/搜工具默认只允许当前窗口文本 `manifestWindow.textPath`、运行级规则快照 `runs/<jobId>/rules/提取规则.md`、当前书籍 reports 根目录（仅 `grep path="reports"` / `grep path="./reports"`）以及本批 `allowedOutputFileNames` 对应报告；裸窗口文件名和 reports/report 文件别名只能映射到这些白名单路径。禁止扫描 `runs/<jobId>` 根、`windows` 目录、`tool-loop-traces`、项目根、绝对路径、其它 bookId reports 或未选中输出名。
- 写工具：`write_file`、`edit_file`、`multi_edit`。
- 写工具只能写当前书籍 `assets/books/<bookId>/reports/`。
- 写入路径只接受业务允许的报告文件名；为兼容模型沿用读工具路径，写工具也接受当前书籍 `assets/books/<bookId>/reports/<outputFileName>`，并在执行前归一化为裸 `outputFileName`。
- 写入路径不得接受其它书籍 ID、未选中输出名、任意项目路径或目录路径。
- 写入路径必须解析真实路径后仍在 reports 根内，防止 `..`、绝对路径、UNC 路径或软链接逃逸。
- 每次写入返回可审计摘要：文件名、操作、变更字节数、预览。
- 写工具正文不得包含运行窗口文件名或内部窗口标识，例如 `window-0001`、`window-0002`、`window-0003`；正式报告应改写为 `窗口 1（第1-5章）`、章节范围、章节名或原文范围。该规则只检查 `write_file.content`、`edit_file.newText` 和 `multi_edit.edits[].newText`，不拦截工具 `path`，也不拦截 `oldText`。
- 写工具正文不得保留 `状态：草案` 或 `状态：模板`；允许写 `状态：原文已复核`、`状态：已抽取` 等非草案状态。
- 只读工具可以并行执行；写工具、未知工具和需要顺序证据的工具必须串行执行。
- 工具结果进入下一轮模型消息前必须套用工具输出预算，超长结果保留头尾并标记截断。

## 点击开始后的详细流程

### 1. Renderer 发起开始

用户点击任务卡“开始”：

```text
Renderer
  -> api.startJob({ jobId })
```

UI 立即做乐观反馈：

- 状态显示“运行中”。
- 进度显示“准备运行”或“窗口 0/N”。
- usage 显示 `输入 0 / 输出 0 / 缓存命中 0`。
- 日志新增“任务已开始，正在准备运行窗口”。
- “开始”按钮隐藏，“暂停”按钮显示。

此时 UI 不能声称模型已经完成请求，也不能显示最终报告。

任务创建或任务参数区必须有 `跳过已提取` 勾选项：

- 默认建议勾选，便于长任务断点续跑和避免重复消耗。
- 勾选后，Runtime 使用进度账本跳过已完成且窗口 hash、模板 hash、模板组 hash、输出文件名 hash 和规则语义 hash 未变化的窗口模板组。
- 取消勾选后，Runtime 不使用账本跳过模型调用；同一本书、同一窗口、同一模板组也会重新进入模型工具循环，重新判断是否补充或修改 Markdown。
- 该选项随任务快照保存，不作为全局默认强制覆盖所有任务。

### 2. Main 校验任务

Main 收到 `jobs:start`：

1. 读取任务记录。
2. 校验状态必须是 `created`、`paused` 或 `blocked_for_user` 这类可运行状态；不再引入独立的 `resume_ready` 状态。
3. 校验已有模型配置和 API key。
4. 校验书籍原文资产和模板快照存在。
5. 校验没有另一个任务正在运行。

成功后：

- 任务状态改为 `running`。
- 写入 `job.started` 事件。
- 立即返回最新 `JobDto` 给 Renderer。
- 后台继续执行 runtime。

失败时：

- 返回 typed error。
- UI 显示失败原因。
- 不进入后台运行。

### 3.1 生成/刷新运行窗口文件

每次任务正式运行前，Runtime 必须先把原文切成本次任务专用的运行窗口文件。上传阶段只保存原文，不再长期保存每章一个 txt 文件。

推荐位置：

```text
<projectRoot>/runs/<jobId>/windows/window-0001.txt
<projectRoot>/runs/<jobId>/windows/window-0002.txt
<projectRoot>/runs/<jobId>/windows/manifest.json
```

切分逻辑参考 CLI 现有实现：

- `src/novel_extractor/chapters.py` 的 `parse_chapters` 和 `build_windows` 是语义基准。
- 桌面端实现可以迁移为 TypeScript 模块，但规则必须保持同等能力：按配置化章节标题正则识别章节，按 `size / stride / max` 生成带重叠的窗口。
- 参数映射为：`size = singleRunChapterCount`，`stride = singleRunChapterCount - overlapChapterCount`；桌面端不直接暴露 CLI 的 `max_windows`，而是先用 `extractionChapterCount` 截断识别章节，再生成窗口。
- 不把章节单独写成资产文件；章节对象只在切分脚本内存和 `manifest.json` 摘要中存在。
- `window-xxxx.txt` 是后续路由、正式提取和读工具可读取的正文输入。

每次开始任务时都要校验 manifest：

- `sourceTextHash`：原文文件内容 hash。
- `splitConfigHash`：`singleRunChapterCount`、`overlapChapterCount`、`extractionChapterCount`、章节标题规则和 splitter 配置 hash。
- `splitterVersion`：切分器版本。
- `windows[]`：窗口 ID、文件名、上下文章节范围、本轮提交章节范围、`windowHash`、字数或字符数。

如果 `manifest.json` 不存在，或以上任一 hash/版本与当前任务快照不一致，Runtime 先删除旧的 `runs/<jobId>/windows/`，再重新生成窗口文件。删除范围必须限制在当前任务运行目录内。

事件：

```text
job.windows.generating
payload:
  jobId
  sourceTextHash
  splitConfigHash
  splitterVersion

job.windows.generated
payload:
  jobId
  windowsRoot
  manifestPath
  sourceTextHash
  splitConfigHash
  splitterVersion
  totalWindowCount
  totalDetectedChapterCount
```

切分失败处理：

- 原文读取、编码、章节识别、窗口参数或文件写入任一步失败，都不进入模型调用。
- 任务状态进入 `blocked_for_user`，UI 可按暂停态显示“继续”，但文案必须说明是“切分失败待处理”。
- 日志记录错误类型、原文文件名、章节标题规则、窗口参数和错误摘要，不持久化完整原文正文。
- 用户修改原文、章节规则或窗口参数后点击继续，Runtime 重新执行本节流程。

UI 刷新：

- 日志显示“正在生成运行窗口文件”。
- 成功后显示“已生成 N 个运行窗口，原文识别 M 章”。
- 失败时显示“运行窗口生成失败”，并展示可操作的错误摘要。

### 3.2 生成提取规则文档

每次任务正式运行前，Main 必须根据本次任务选择的模板快照生成运行级不可变规则快照，并同步项目级 latest copy。

运行级规则快照位置：

```text
<projectRoot>/runs/<jobId>/rules/提取规则.md
```

项目级 latest copy / 人工入口位置：

```text
<projectRoot>/rules/提取规则.md
```

`runs/<jobId>/rules/提取规则.md` 是本次运行的审计依据，生成后不可变；后续继续、恢复和日志回放都引用该路径。`rules/提取规则.md` 每次任务开始前会被覆盖为 latest copy，便于人工查看和编辑入口展示，但不能作为历史任务的唯一追溯依据。

运行上下文必须同时保存：

```text
rulesSnapshotPath: runs/<jobId>/rules/提取规则.md
rulesLatestPath: rules/提取规则.md
rulesDocumentHash
rulesSemanticHash
```

`rulesDocumentHash` 计算完整 Markdown 文档，包含 `jobId`、生成时间、路径等审计元数据，用于日志追溯和变更检测。`rulesSemanticHash` 计算排除易变元数据后的规范化规则语义内容，用于进度账本跳过判断。账本不得使用 `rulesDocumentHash`，否则同一语义规则会因为运行时间或 `jobId` 不同而误判为不可跳过。

`提取规则.md` 主要服务模板路由，同时也作为审计、账本跳过和人工排错依据。Runtime 不把它当成关键词硬匹配表；正式提取仍以任务快照中的完整模板正文为准。

生成内容参考 CLI 当前规则来源：

- 模板目录 `README.md` 的通用写作规则、写全规则、禁止事项和模板清单。
- CLI 配置 `config/novel_extractor.mvp.yaml` 的 `templates[].card`、`output_filename` 和 `template_groups`。
- 本次任务选择的模板快照，包括模板名称、模板正文摘要、输出文档名和用户配置的路由说明。

生成优先级：

1. 显式 `routeDescription` / `routeCard` 优先作为路由说明。
2. 旧 CLI `templates[].card` 可迁移为路由说明。
3. 没有路由说明时，按模板名、输出文件名和模板正文标题/前段生成稳定摘要，不另调模型生成。
4. 有显式模板组时按模板组生成路由规则；没有模板组时，优先按输出文件名成组，再退化为一模板一组。

规则文档至少包含：

```md
# 提取规则

> 项目：<项目名>
> 任务：<jobId>
> 生成时间：<ISO 时间>
> 规则来源：模板快照、项目模板库、CLI 模板目录规则

## 通用抽取规则

## 模板路由规则

| 模板组 | 模板 | 输出文档 | 路由说明 | 不确定时策略 |

## 模板组快照

## 路由模型返回格式

## 写入规则

只写工具使用约束摘要，不写完整工具 JSON Schema。完整工具 schema 由 Runtime 在正式提取模型调用时通过 `tools` 参数发送。

## 跳过已提取策略
```

事件：

```text
job.rules.generated
payload:
  jobId
  rulesSnapshotPath
  rulesLatestPath
  rulesDocumentHash
  rulesSemanticHash
  selectedTemplateCount
  templateGroups
```

### 6. 账本预检查和模板路由

Runtime 在每个窗口开始后先做账本预检查，再决定是否需要大模型模板路由。模板快照和运行级 `提取规则.md` 是路由模型的输入依据，Runtime 不按关键词、模板名称或硬编码规则直接决定命中。

执行顺序：

1. 从模板组快照得到当前窗口的候选模板组。
2. 如果 `skipAlreadyExtracted = true`，Runtime 先按账本 key 查询每个窗口-模板组。
3. 账本命中 `completed` 或 `no-update` 的模板组，直接记录 `job.template.skipped`，该组不再进入路由模型调用，也不进入正式提取模型调用。
4. 如果所有候选模板组都已被账本跳过，当前窗口直接完成，不发起路由调用。
5. 对未跳过的候选模板组，先读取同一运行快照下可用的路由缓存；缓存的 `windowHash`、`templateGroupHash`、`outputFileNamesHash` 和 `rulesSemanticHash` 必须匹配。
6. 无可用路由缓存或缓存 hash 不匹配时，才对剩余候选模板组执行大模型模板路由。

路由模型输入：

- 当前窗口文件正文，即 `window-xxxx.txt` 内容。
- 本轮提交章节范围。
- 本次任务生成的运行级 `提取规则.md` 或其结构化摘要。
- 本次任务的模板快照摘要。
- 本次任务的模板组快照摘要。

路由模型输出必须是简单结构化 JSON。模型只返回命中的模板组 ID，以及跳过原因：

```json
{
  "matchedGroupIds": ["resource_group", "faction_group"],
  "skipped": [
    {
      "id": "npc_group",
      "reason": "本窗口没有角色性格变化"
    }
  ]
}
```

Runtime 根据 `matchedGroupIds` 从模板组快照展开 `templateIds` 和 `outputFileNames`。路由模型不得返回模板 ID、输出文件名、confidence 或其它执行字段。

Runtime 校验：

- `matchedGroupIds` 必须是字符串数组。
- 每个 `groupId` 必须来自本次模板组快照。
- `skipped[].id` 必须来自本次候选模板组 ID。
- 如果 `matchedGroupIds` 为空，`skipped` 必须说明每个候选组为什么跳过。
- `matchedGroupIds = []` 不是失败；只要 `skipped` 覆盖所有候选组并写明原因，就表示当前窗口正常无命中。
- 结构化 JSON 解析失败或引用非法 ID 时，先按路由错误策略处理。
- 如果路由模型不确定，应保守命中，而不是跳过；如果 `skipped.reason` 出现“不确定”“可能相关”等表达，Runtime 要求重试或按保守命中处理。

路由错误策略走配置化 `RouteFailurePolicy`，并随运行时策略快照冻结：

```ts
interface RouteFailurePolicy {
  maxRetries: number;
  fallbackStrategy: "none" | "semanticRuleFilter" | "matchAll";
  fallbackSource: "runtimePolicySnapshot" | "rulesSnapshot";
  onFallbackNoMatch: "no-update" | "blocked_for_user";
}
```

默认策略建议为：`maxRetries = 1`、`fallbackStrategy = "semanticRuleFilter"`、`fallbackSource = "rulesSnapshot"`、`onFallbackNoMatch = "no-update"`。fallback 规则必须来自任务规则快照或运行时策略快照中的配置化规则，不允许运行时在代码里硬编码关键词。路由 JSON 不合法、`matchedGroupIds` 非数组、非法 `groupId`、`skipped` 缺 reason 或不确定跳过时，按 `maxRetries` 重试；重试耗尽后按 `fallbackStrategy` 执行。fallback 命中则继续本窗口并在日志中标明来源；fallback 无命中时按 `onFallbackNoMatch` 进入 `no-update` 或 `blocked_for_user`。路由模型正常返回空命中和 fallback 空命中都不运行模板组，但日志必须区分来源。

路由调用自身进入 usage 和日志，不并入后续正式提取调用中模糊处理。路由调用不发送工具列表；工具 schema 只在命中模板组后的正式提取工具循环中发送。

事件：

```text
job.route.model.call.started
job.route.model.call.completed
job.templates.routed
payload:
  jobId
  windowId
  routeCallIndex
  rulesDocumentHash
  rulesSemanticHash
  matchedGroupIds
  expandedGroups:
    - groupId
      templateIds
      outputFileNames
  skipped:
    - id
      reason
```

UI 刷新：

- 日志显示“命中模板：丹药分析、势力分析”。
- 未命中时日志显示“路由完成：未命中模板组”，并逐条显示中文模板组名和跳过原因，例如“跳过「神兵利刃分析」：本窗口没有出现神兵、法宝、武器炼制或战斗使用信息”。
- 用户可见日志必须显示模板组中文名或模板中文名，不显示 `weapon_group`、`resource_group` 等内部 ID。内部 ID 只用于 Runtime 校验、事件 payload 和账本。
- 日志显示路由模型本轮 usage：输入、输出、缓存命中。
- 如果窗口没有命中任何模板，窗口直接完成，状态为 `no-update`。
- 如果所有候选模板组在账本预检查中已跳过，日志显示“窗口已由进度账本跳过，无需路由调用”，且本窗口不产生路由模型 usage。

### 7. 进度账本记录和跳过规则

账本跳过的主粒度是“运行窗口文件 + 模板组快照 + 输出文件集合”。窗口编号只是定位信息，必须和 `windowHash` 一起使用，不能只按 `window-0001` 这类序号判断。

对每个窗口和模板组计算账本 key：

```text
bookId
windowId
windowHash
templateGroupId
templateGroupHash
outputFileNamesHash
rulesSemanticHash
```

附加审计字段：

```text
textPath
sourceTextHash
splitConfigHash
submittedChapterRange
contextChapterRange
routeResultHash
```

`windowHash` 计算 `window-xxxx.txt` 的完整内容。只要原文、章节识别规则、窗口参数或重叠上下文导致窗口文件变化，本窗口就不会命中旧账本跳过。`sourceTextHash` 和 `splitConfigHash` 作为审计字段，帮助解释为什么同一任务继续时重新生成了窗口。

只有当任务参数 `skipAlreadyExtracted = true` 时，才执行账本跳过检查。该检查发生在路由模型调用之前，并按候选模板组逐个判断。

若勾选 `跳过已提取`，并且账本中同一本书、同一个窗口文件 hash、同一模板组快照、同一组输出文件和同一 `rulesSemanticHash` 已经 `completed` 或 `no-update`：

- 跳过该窗口-模板组的全部模型调用，包括路由调用和正式提取工具循环调用。
- 记录 `job.template.skipped`。
- UI 日志显示“已完成且窗口文件、模板组和输出文件未变化，跳过”。

账本记录必须包含每个输出文件的状态：

```text
status: completed | no-update | failed
files:
  - outputFileName
    status: updated | no-update | skipped | failed
    beforeHash
    afterHash
```

只有模板组里的预期输出文件全部 `updated`、`no-update` 或 `skipped`，整个模板组才可记录为 `completed` 或 `no-update`。任一输出文件失败，则该模板组账本状态为 `failed`，后续不能被跳过。

若未勾选 `跳过已提取`：

- 不读取账本做跳过决策。
- 仍记录当前窗口和模板组的 hash，便于本次任务结束后更新账本。
- UI 日志显示“跳过已提取未启用，本轮重新提取”。

这只影响是否跳过模型调用，不改变“重复完整则跳过，不完整则补充或修改”的写入规则。即使重新提取，模型仍必须读取已有 Markdown 并自行判断是否有必要写入。

### 8.1 模板组拆批策略

命中的模板组按 `maxFullTemplatesPerCall` 和 prompt budget 拆成一个或多个模型工具循环子批次；默认 `maxFullTemplatesPerCall = 1` 时，一次工具循环只处理一个完整模板。拆批只拆模板组里的模板，不拆当前运行窗口；窗口文件的上下文范围和本轮提交范围保持不变。

拆批顺序：

1. 先按输出 Markdown 文件聚合相关模板。
2. 聚合后仍超过 `maxFullTemplatesPerCall` 或 prompt budget 时，按模板快照顺序拆批。
3. 默认 `maxFullTemplatesPerCall = 1` 时，每个完整模板独立进入一个子批次。
4. 单个模板本身都放不进 prompt budget 时，当前窗口模板组失败，提示用户调小 `singleRunChapterCount`、降低 `overlapChapterCount` 或更换更大上下文模型。

按输出文件拆批的目标是减少多个批次同时修改同一文件的风险。例如：

```text
批次 1：丹药分析.md 相关模板
批次 2：势力分析.md 相关模板
批次 3：功法分析.md 相关模板
```

如果单个输出文件仍然太大，模板不可切半，只按模板顺序拆：

```text
丹药分析.md 批次 1：模板 1-3
丹药分析.md 批次 2：模板 4-6
丹药分析.md 批次 3：模板 7-8
```

每个子批次都是独立工具循环：

- 必须先查询自己允许写入的目标 Markdown。
- 独立记录 usage、日志、工具调用和文件状态。
- 子批次串行执行，不并发写同一个报告。
- 后续子批次必须读取前序子批次已经写过的 Markdown，再决定补充、修改或 `NO_UPDATE`。

账本仍汇总到原模板组。子批次只是执行细节：

- 全部子批次成功且至少一个文件更新：模板组 `completed`。
- 全部子批次成功且都无更新：模板组 `no-update`。
- 任一子批次失败：模板组 `failed`。

### 9. Prompt 构造

System prompt 放稳定内容，用于缓存：

- 固定任务规则。
- 写入协议。
- 工具使用规则。
- 当前模板组的模板正文，以输出文件名作为标题。

User prompt 放窗口变化内容：

- 小说名。
- 窗口 ID。
- 上下文章节范围。
- 本轮提交章节范围。
- 本轮允许写入的 Markdown 文件名。
- 当前 reports 目录相对路径。
- 当前窗口文件正文。

User prompt 不预塞旧报告片段。旧报告是否已有相关内容、哪里需要补充或修改，必须由模型在工具循环中使用 `read_file` 或 `grep` 自己查询。

关键写入规则必须进入 prompt：

- 按模板要求写入对应 Markdown。
- 写入前必须 `grep` 或 `read_file` 查询目标文档。
- 重复且完整则跳过。
- 重复但不完整则补充或修改。
- 新事实必须来自当前窗口文件中的原文内容。
- 禁止使用模型对作品全书、后续章节、未读窗口或常识剧情的先验知识；只能写当前窗口文本明示或当前已有报告已证实的事实；涉及未来真相、真实身份、夺舍、寿元、后续影响等当前窗口未说明内容，必须写原文未说明或不写。
- 资料来源、参考范围、更新日期等元信息只能根据实际使用的当前窗口、已读取报告和当前运行日期填写；不得遗漏已使用窗口、不得声称未读取来源，更新日期不得晚于当前运行日期。
- 材料分析类或资源类模板中，未命名但能由原文稳定描述的材料、药草、药汁、药物、灵液、资源产出源也应记录，不得仅因没有专名或呈现为成品形态就直接 `NO_UPDATE`；特殊容器或物件若在当前窗口明示为资源产出源、材料载体或关键性质/功能载体，也应作为资源/产出源记录，未知效果或用途写“原文未说明”。
- 正式报告的资料来源、参考范围等公开元数据只能写窗口编号、章节范围、章节名或原文范围；不得写 `runs/<jobId>/...`、`assets/books/...`、本机绝对路径、AppData 项目路径等内部运行/项目路径，也不得写“后续窗口”等流程性措辞。
- 没有任何更新时最终返回精确 `NO_UPDATE`。

#### Prompt budget

每轮模型请求前 Runtime 计算 prompt budget：

- 统计 system prompt、user prompt、历史 assistant/tool messages、工具 schema 的估算 token。
- 工具输出预算不等于 prompt budget；工具结果先按工具预算截断，再进入下一轮 prompt。
- 配置项包括 `contextWindow`、`warnRatio`、`hardRatio`、`strategy`，并写入任务快照。
- 达到 `warnRatio` 时任务继续，日志记录“prompt 接近上下文窗口”。
- 达到 `hardRatio` 时不静默裁剪窗口正文；当前窗口失败，并提示用户降低单次运行章节数、降低重叠章节数或调整模型上下文窗口。

预算估算可以沿用 CLI 的保守估算方式：英文按约 4 字符 1 token，中文按字符数兜底。实现时允许按模型适配器替换更准确的 tokenizer，但不能影响事件字段和 UI 展示。

### 12. 工具调用

如果模型返回 tool calls：

1. Runtime 先按模型返回顺序记录所有工具 dispatch，日志能立即看到模型准备调用哪些工具。
2. 连续的已知只读工具可以并行执行，默认最大并行数走配置。
3. 写工具、未知工具、权限受限工具必须串行执行，保持模型返回顺序。
4. 工具结果事件仍按模型返回顺序追加，即使内部只读工具并行完成，也不打乱日志。
5. 工具结果作为 tool message 返回给模型，进入下一轮模型调用。
6. 未知工具不直接崩溃，返回 `error: unknown tool <name>` 给模型，由模型自行修正。

事件：

```text
job.tool.call.dispatched
job.tool.call.started
job.tool.call.completed
job.tool.call.failed
job.tool.result.truncated
```

完成事件 payload：

```text
jobId
windowId
groupId
toolName
argumentsSummary
resultSummary
success
readOnly
durationMs
truncated
```

UI 刷新：

- 展开日志显示工具名称和摘要。
- `read_file` / `grep` 显示读取或命中数量。
- 写工具显示写入文件和变更摘要。
- 工具参数、工具结果、错误详情默认显示摘要，允许展开查看脱敏详情。

工具输出预算：

- `read_file` 默认返回最多 2000 行，支持 offset/limit 分页。
- `grep` 默认最多 200 条命中，超出后提示截断。
- `glob` 默认最多 1000 条结果，超出后提示截断。
- 单次工具结果进入模型前默认最多 32KB。
- 超过单次工具结果预算时保留头部和尾部，中间插入截断说明，日志追加 `job.tool.result.truncated`。
- 工具预算、tool loop 最大轮次和超时走配置并进入 `RuntimePolicySnapshotDto` / `ToolRuntimeSnapshotDto`；同一任务暂停、继续、重跑时保持一致。

工具错误处理：

- 可恢复错误返回给模型，例如替换文本不存在、grep 无匹配、read_file 文件不存在。
- 工具边界、reports 根异常、认证失败等不可由模型自行修复的错误会记录日志并进入 `blocked_for_user`，等待用户处理后继续。
- 用户取消属于主动终止，不自动继续。
- 工具参数 JSON 非法时，把错误和对应工具 schema 摘要返回给模型，允许模型重发合法参数。

### 13. Markdown 写入

写入规则：

- 每个模板对应一个输出 Markdown。
- 输出文件名优先使用模板快照中的 `outputFileName`。
- 如果没有显式输出名，模板名去掉末尾“模板”并加 `.md`。
- 如果模板名不含“模板”，直接使用模板名加 `.md`。
- 文件名必须是单文件名，不允许路径分隔符、绝对路径、UNC 路径或 `..`。
- 文件名 sanitizer、sanitizer 版本、大小写/Unicode 归一化和同名冲突策略必须来自运行策略快照或项目配置；不允许在写入分支中临时硬编码替换规则。

命中“丹药分析模板”和“势力分析模板”时，最终应生成或更新：

```text
assets/books/<bookId>/reports/丹药分析.md
assets/books/<bookId>/reports/势力分析.md
```

写入模式：

- 新文件：`write_file` 创建完整 Markdown。
- 写工具 `path` 可以使用裸输出文件名，或当前书籍的 `assets/books/<bookId>/reports/<outputFileName>`；Runtime 执行前必须归一化为裸输出文件名。
- 既有文件：`write_file` 默认不允许覆盖；Runtime 必须把“已有报告不能用 `write_file` 覆盖，需先 `read_file`/`grep` 再 `edit_file`/`multi_edit`”作为可恢复 tool result 回放给模型。若本轮已经成功查询同一报告，且 `write_file.content` 完整包含当前磁盘旧内容，可作为安全重写放行。
- 旧文件存在且需要追加：`edit_file` 或 `multi_edit` 按精确文本补充；跨窗口、新 tool loop 或运行前已存在旧报告必须先查询同一报告，同一 tool loop 内新建成功的本批允许报告可在后续轮次直接编辑修正。
- 旧文件存在且完整重复：不写入，最终返回 `NO_UPDATE` 或只完成其它文件写入。
- 写工具正文校验只检查 `write_file.content`、`edit_file.newText` 和 `multi_edit.edits[].newText`；不得拦截 `oldText`，以便模型删除旧报告中的脏路径；不得拦截工具 `path` 参数本身，因为它属于受控协议路径。
- `write_file.content`、`edit_file.newText` 或任一 `multi_edit.edits[].newText` 含 `runs/job-`、`runs/<jobId>/windows`、`assets/books/`、Windows 绝对路径、`AppData\Roaming\@novel-extractor` 或“后续窗口”等内部路径/流程元数据时，Runtime 返回 recoverable tool error，提示改写为窗口编号、章节范围、章节名或原文范围；不得写磁盘、不得注册或更新 `ReportAsset`，也不得计为本窗口成功写入。
- `write_file.content`、`edit_file.newText` 或任一 `multi_edit.edits[].newText` 含 `window-0001` 这类运行窗口文件名/内部窗口标识时，Runtime 返回 recoverable tool error，提示改写为 `窗口 1（第1-5章）`、章节范围、章节名或原文范围；不得写磁盘、不得注册或更新 `ReportAsset`，也不得计为本窗口成功写入。
- `write_file.content`、`edit_file.newText` 或任一 `multi_edit.edits[].newText` 含 `状态：草案` 或 `状态：模板` 时，Runtime 返回 recoverable tool error；允许模型改为 `状态：原文已复核`、`状态：已抽取` 等非草案状态，或删除状态行。
- Prompt 必须明确声明模板示例、字段说明、示例事件链和通用体系词只作为格式参考；`修仙世界` / `修仙界`、`法修` / `武修`、`灵石` / `灵草` / `矿产` 等词不得因为出现在模板中就写入正式报告，只有当前窗口原文或已读取既有报告明确证实时才可写。`长期余波`、`可参考点` 等分析字段不得用模板泛化话术推导未来影响，当前窗口未说明的后续影响必须写原文未说明或不写。
- 被上述保护拦截的 `write_file` 不得修改磁盘文件、不得注册或更新 `ReportAsset`，也不得计为本窗口成功写入。

写入完成后发布：

```text
job.file.written
payload:
  jobId
  windowId
  groupId
  reportFileName
  reportId
  operation
  changedBytes
  preview
```

UI 刷新：

- 任务卡不需要塞完整内容。
- 展开日志显示“写入：丹药分析.md”。
- 任务详情的写入文件列表新增或更新该文件。
- 资产页报告列表可在当前书籍选中时刷新。

### 14. 工具循环完成校验

`NO_UPDATE` 是“模板组本轮没有任何文件需要更新”的最终状态，不是单个文件的自由文本说明。

模型工具循环结束后，Runtime 校验：

- 写入文件必须属于本轮允许的输出文档列表。
- 写入前必须查询过目标文档，使用 `read_file` 或 `grep`。
- 如果配置要求证据章节，则新增内容必须包含当前窗口章节证据。
- 写入后的 Markdown 必须能被基础预览器读取。

合法结束只有两种：

1. 至少一次合法写入，并且模板组内所有预期输出文件都有文件级状态。
2. 零写入，并且最终文本精确等于 `NO_UPDATE`，且模板组内所有预期输出文件都已被 `read_file` 或 `grep` 查询过。

文件级状态规则：

- 已写入的文件记录 `updated`。
- 未写入但已查询且模型判断无需更新的文件记录 `no-update`。
- 未查询的预期输出文件不能自动记为 `no-update`。
- 如果模型只处理了部分预期输出文件，Runtime 要求继续工具循环或让当前模板组失败。

如果已经写过一个或多个文件，最终回答不再要求是 `NO_UPDATE`。Runtime 以工具写入事件和文件级状态为准；最终文本可以是简短完成说明或结构化 summary。

校验失败时：

- 当前窗口模板组标记 failed。
- 任务进入 `blocked_for_user`，等待用户调整模板、报告文件、章节参数、工具配置或模型配置后继续。
- UI 显示具体原因，不吞掉错误。

### 15. 模板组完成

完成后写进度账本：

- 有写入，且所有预期输出文件都有状态：模板组 `completed`。
- 无写入，最终文本精确 `NO_UPDATE`，且所有预期输出文件已查询：模板组 `no-update`。
- 失败：`failed`，记录错误摘要。

模板组账本状态由文件级状态汇总而来。不能因为某个文件没有写入，就自动认为它 `no-update`；必须有查询证据。

事件：

```text
job.template.completed
payload:
  jobId
  windowId
  groupId
  writtenFiles
  noUpdate
```

UI 刷新：

- 日志显示“模板组完成，写入：丹药分析.md、势力分析.md”。
- 若无更新，显示“无新增内容”。

如果用户在窗口运行期间点击暂停：

- 当前模型请求和当前窗口不强制中断。
- 窗口完成后进入 `paused`。
- UI 显示“已暂停”，按钮变为“继续”“删除任务”。

## UI 显示规范

### 任务卡紧凑信息

任务卡默认展示：

```text
状态
进度
输入 X / 输出 Y / 缓存命中 Z
```

不展示费用。

### 展开日志

展开后展示：

- 任务开始。
- 窗口开始/完成。
- 命中模板。
- 模型调用开始。
- usage 更新。
- 工具调用摘要。
- 文件写入摘要。
- 跳过原因。
- 错误详情。

日志要尽可能详细，作为任务运行轨迹，而不是只显示摘要：

- 每次大模型请求都要显示：模型、窗口、模板组、调用序号、消息摘要、工具 schema 摘要。
- 每次大模型响应都要显示：是否返回工具调用、最终文本摘要、usage、缓存命中。
- 每次工具调用都要显示：工具名、参数摘要、开始时间、结束时间、成功/失败、结果摘要。
- 每次写入都要显示：文件名、操作类型、变更字节数、预览片段。
- 每次错误都要显示：错误阶段、错误码、脱敏后的错误消息、是否可恢复。
- 长 prompt、长响应和长工具结果默认折叠显示摘要；持久化日志默认不长期保存完整 prompt、窗口正文或完整工具结果。

日志按事件时间顺序从上到下追加，第一条在最上面，用户向下滚动查看后续事件。日志区域必须可滚动，不能因为日志过长挤压任务卡或页面布局。

日志默认持久化结构化事件、hash、文件名、usage 和错误摘要。调试日志默认开启，但默认开启只表示保存结构化调试轨迹，不表示长期保存完整小说正文、完整 prompt 或完整工具结果。

日志显示名规则：

- 用户可见日志、任务卡和 UI 文案显示 `groupDisplayName`、模板中文名和 Markdown 文件名。
- `groupId`、`templateId`、`toolSchemaHash` 等内部 ID 默认只进入事件 payload、调试详情和账本，不出现在普通日志行里。
- 如果错误来自非法 ID，普通日志也要同时给出中文候选项，例如“合法模板组：神兵利刃分析、丹药分析、势力分析”。

日志脱敏规则：

- API key、Authorization header、凭据引用和本地隐私路径按现有脱敏规则处理。
- 可以展示小说章节片段、模板片段和 Markdown 预览，但长文本需要折叠。
- 持久化日志默认只保存摘要、hash 和必要定位信息，避免把本地小说正文作为长期日志内容保存。
- 不显示外部模型的隐藏推理内容；只显示 API 可见的请求、响应、工具调用和工具结果。

### 写入文件列表

### Hash 组成和归一化

账本相关 hash 必须使用稳定序列化，统一 UTF-8、LF 换行、对象字段固定排序、数组排序规则明确、路径分隔符规范化为 `/`。所有 hash 都不包含运行时间、`jobId`、latest 路径、日志文案等易变元数据。

`templateHash`：

- 字段顺序：`templateId`、`templateName`、`templateBody`、`outputFileName`、`routeCard`、`routeDescription`、`groupId`。
- `templateBody` 保留正文语义，换行归一为 LF；不包含 `createdAt`。
- `outputFileName` 先经过文件名 sanitizer 归一化，再参与 hash。

`groupHash`：

- 字段顺序：`groupId`、`groupDisplayName`、`templateIds`、`maxFullTemplatesPerCall`。
- `templateIds` 使用模板组快照中的稳定顺序；该顺序影响 prompt 和拆批，不可临时排序。
- 若模板组显示名只影响 UI、不影响路由语义，可由实现配置决定是否纳入，但同一项目内必须固定。

`outputFileNamesHash`：

- 来源是模板组展开后的预期输出文件名集合。
- 文件名先做 Unicode NFC、trim、路径分隔符拒绝、Windows 保留名拒绝和扩展名归一，再按规范化文件名升序去重。
- hash 输入包含 sanitizer 版本和冲突策略，避免文件名规则变化后误跳过。

`rulesSemanticHash`：

- 来源是规则快照中的语义内容：模板路由规则、模板组快照摘要、输出文件约束、通用抽取规则、写入规则、`RouteFailurePolicy` 中影响命中的字段。
- 排除 `jobId`、生成时间、`rulesSnapshotPath`、`rulesLatestPath`、日志展示文本和其它审计元数据。
- 模板组按 `groupId` 稳定排序；组内模板按模板组快照顺序；正文换行统一 LF。

`rulesDocumentHash`：

- 计算完整 `runs/<jobId>/rules/提取规则.md` 文件内容，包含审计元数据。
- 只用于审计、日志和继续前变更提示，不参与账本跳过 key。

## 状态机

P0 任务状态：

```text
created
  -> running
running
  -> pause_requested
  -> completed
  -> blocked_for_user
  -> failed
pause_requested
  -> paused
paused
  -> running
  -> deleted
blocked_for_user
  -> running
  -> deleted
failed
  -> deleted
completed
  -> deleted
deleted
```

`paused` 表示用户主动暂停；`blocked_for_user` 表示运行中遇到需要用户处理的错误。两者 UI 都可以显示“继续”，但日志和状态文案必须区分来源。

错误分类：

- recoverable 错误进入 `blocked_for_user`，包括认证和模型配置错误、运行窗口切分错误、路由 fallback 无命中且策略要求人工处理、tool loop 最大轮次/超时、工具循环校验失败、模型协议错误、写入边界错误、prompt hard 超限、报告文件冲突和旧缓存/状态读取异常。
- fatal 错误才进入 `failed`，仅用于用户无法通过修改输入、配置、模板或报告文件恢复的任务记录损坏、运行快照不可解析且无可用审计信息、账本存储不可写且无法安全暂停等场景。
- 模板组账本状态里的 `failed` 只表示该窗口-模板组不可被账本跳过，不等同于任务状态 `failed`；任务状态通常应进入 `blocked_for_user`。
- 不再定义 `resume_ready`。崩溃恢复、应用重启后的手动继续能力由 `blocked_for_user` 或 `paused` 表示。

应用退出、崩溃或进程被杀：

- 原 `running` 任务恢复为 `blocked_for_user`，原 `pause_requested` 任务恢复为 `paused`，不得自动继续。
- UI 提示“上次运行中断，请手动继续”。
- 继续时若任务勾选了 `跳过已提取`，依赖进度账本跳过已完成窗口模板组；若未勾选，则继续后仍重新提取未完成后的窗口模板组。

## Usage 刷新策略

usage 更新只能在模型返回 usage 后发生。

刷新点：

- 路由调用返回 usage 后发布 `job.usage.updated`。
- 正式提取调用每轮返回 usage 后发布 `job.usage.updated`。
- 每个运行窗口完成后在 `job.window.completed` 附带累计 usage。
- 任务完成后在 `job.completed` 附带最终 usage。

显示格式：

```text
输入 12,345 / 输出 2,345 / 缓存命中 9,876
```

如果 `cacheHitTokens = 0` 也正常显示 0。不要使用“费用 0”占位。

## 重复、补充和修改规则

模型层规则：

- 写入前必须读取或搜索目标 Markdown。
- 若发现同名条目完整，跳过。
- 若同名条目不完整，补充字段或替换局部段落。
- 若新章节提供纠错信息，可修改旧条目，但必须注明证据章节。
- 不允许凭空补全。

Runtime 层强制规则：

- 没有查询目标文档就写入，失败。
- 写入非本轮允许的报告文件，失败。
- 写入项目外路径，失败。
- 缺少必要证据章节，失败。

这些规则避免完全依赖 prompt 约束。

## 错误处理

### 认证和模型配置错误

例如 API key 无效、模型不存在、HTTP 401：

- 任务进入 `blocked_for_user`，不继续后续窗口。
- UI 显示明确错误。
- 日志脱敏 API key。
- 用户修正模型配置后可以点击继续，从失败窗口继续运行。

### 运行窗口切分错误

例如原文读取失败、章节标题规则无法识别、窗口参数非法、写入 `window-xxxx.txt` 失败或 manifest 校验失败：

- 不进入路由模型和正式提取模型调用。
- 任务状态进入 `blocked_for_user`，UI 以待处理/暂停态展示。
- 日志记录切分阶段、原文文件名、章节标题规则、窗口参数、错误摘要和是否已清理旧窗口目录。
- 用户修正原文、章节规则或窗口参数后可以点击继续，继续时重新生成运行窗口文件。

### 工具可恢复错误

例如 `grep` 无结果、`edit_file` 找不到 oldText：

- 工具结果返回给模型。
- 模型可改用 `write_file` 或其它策略。
- 若工具循环最终仍无法产出合法写入或 `NO_UPDATE`，当前模板组失败，任务进入 `blocked_for_user`，等待用户处理后继续。

### 写入边界错误

例如路径逃逸、绝对路径、reports 根异常：

- 任务进入 `blocked_for_user`。
- 不把错误返回给模型继续尝试。
- 日志记录具体边界原因和被拒绝路径的脱敏摘要。

### 模型协议错误

例如模型不调用工具也不返回 `NO_UPDATE`：

- 当前窗口模板组失败。
- 任务进入 `blocked_for_user`。
- UI 显示“模型没有按写入协议操作”。
- 用户修改模板、模型配置、章节参数或报告文件后可以继续；继续时从失败窗口或失败模板组恢复，已成功账本仍可跳过。

## 暂停和继续

暂停是窗口边界暂停：

1. 用户点击“暂停”。
2. 任务状态变为 `pause_requested`。
3. UI 显示“将在当前窗口后暂停”。
4. 当前窗口继续完成。
5. 窗口完成后状态变为 `paused`。

继续：

1. 用户点击“继续”。
2. Runtime 先检测关键配置、模板和规则文件是否相对当前运行快照发生变化。
3. 若没有关键变化，Runtime 从失败窗口或下一个未完成窗口模板组继续。
4. 若存在关键变化，UI 弹窗列出变化项，并询问是否使用新规则继续。

继续前需要检测的关键变化：

- 模板正文。
- 模板路由说明。
- 模板组配置。
- 任务选择的模板列表。
- `singleRunChapterCount`、`overlapChapterCount`、`extractionChapterCount`。
- 模型 ID。
- 项目级 `提取规则.md`。
- 工具配置、prompt budget 配置。

弹窗示例：

```text
检测到提取规则已变化

以下内容和任务开始时不同：
- 模板「神兵利刃分析」正文已修改
- 模板组「资源分析」配置已修改
- 章节参数：单次运行章节数 5 -> 4
- 模型：DeepSeek V4 -> DeepSeek V4.1

是否使用新规则继续？
[使用新规则继续] [取消]
```

如果用户选择“使用新规则继续”：

- 生成新的运行快照，包括模板快照、模板组快照、规则 hash、工具快照和 prompt budget 快照。
- 从失败窗口或失败模板组继续。
- 已成功窗口不自动重跑。
- 后续窗口使用新规则。
- 旧账本不会被新规则误判为已完成，因为新的 `templateHash`、`groupHash`、`rulesSemanticHash` 或参数 hash 会参与后续判断。
- 日志记录“用户确认使用新规则继续”，并记录新旧快照 hash。

如果用户选择“取消”：

- 任务保持 `blocked_for_user` 或 `paused`。
- 不继续运行。

若任务勾选 `跳过已提取`，已完成且 hash 未变化的窗口模板组通过账本跳过；若未勾选，继续运行也不因旧账本跳过。

## 报告资产注册

每次 `job.file.written` 后都要确保报告资产存在：

- 如果同一本书已有同名报告，更新 `updatedAt`、`byteSize`、路径映射。
- 如果没有同名报告，创建新 report id。
- 任务记录维护 `resultReportIds` 集合。

报告文件是书籍资产，不是任务私有文件。多个任务可能共同更新同一个 Markdown。

## 与 CLI 的对齐点

桌面端必须保留 CLI 的这些语义：

- 运行窗口逐个处理。
- 任务运行前生成或读取可追溯的提取规则来源。
- 大模型模板路由命中后只处理相关模板组。
- System prompt 放模板和稳定规则，User prompt 放运行窗口内容。
- 写入前查旧报告。
- 重复完整跳过，不完整补充或修改。
- `NO_UPDATE` 表示无写入。
- usage 每次模型响应后记录。
- 进度账本按小说、窗口、模板组、窗口 hash、模板 hash 判断跳过。

桌面端需要改进 CLI 的这些点：

- 使用结构化事件和 DTO，而不是控制台文本作为状态来源。
- usage DTO 明确包含缓存命中，不把费用带进 UI。
- 任务开始 IPC 不阻塞到任务结束。
- 多报告结果以列表表示，不再只保存一个 `resultReportId`。
- 文件写入事件直接驱动资产页刷新。
- 工具系统参考 Reasonix：稳定工具 schema、任务级工具快照、只读并行、写入串行、工具输出预算和写入根限制。

### 单元测试

- 模板输出名：`丹药分析模板` -> `丹药分析.md`，`势力分析模板` -> `势力分析.md`，显式输出名优先。
- 提取规则文档：任务开始前根据选择模板生成运行级规则快照 `runs/<jobId>/rules/提取规则.md`，并同步项目级 latest `rules/提取规则.md`；文档包含模板组、路由说明、不确定时策略、输出文件和写入规则。
- 运行窗口：任务开始时按 CLI 章节识别和窗口生成语义切出 `window-xxxx.txt`，重叠章节只做上下文，不作为本轮提交章节。
- 大模型模板路由：每个窗口先产生一次结构化路由结果，usage 和日志单独记录。
- 路由结果校验：`matchedGroupIds` 非数组、非法 `groupId`、`skipped` 缺少必要原因或 JSON 解析失败时按模型错误处理；`templateIds` 和输出文件名由 Runtime 从快照展开。
- 路由空命中：`matchedGroupIds = []` 且 `skipped` 覆盖所有候选组并写明原因时，窗口正常 `no-update`，不是失败。
- 路由失败兜底：路由模型重试后仍失败时按 `RouteFailurePolicy` 执行配置化 fallback；fallback 规则必须来自规则快照或运行策略快照，不允许运行时硬编码关键词。
- 路由日志显示名：用户可见日志显示中文模板组名，例如“神兵利刃分析”，不显示 `weapon_group`、`resource_group` 等内部 ID。
- 路由保守命中：模型不确定时必须命中并写明 reason，不能直接跳过。
- 模板快照：任务创建时保存模板快照和模板组快照，任务继续和重跑不读取后续编辑后的模板。
- 模板组拆批：按 `maxFullTemplatesPerCall` 和 prompt budget 形成工具循环子批次，优先按输出文件聚合，同一输出文件仍过大时按模板顺序拆批；默认值为 1 时每个完整模板独立执行。
- 子批次执行：子批次串行执行，后续子批次必须读取前序写入后的 Markdown；批次状态最终汇总到原模板组账本。
- 单模板过大：单个模板无法放入 prompt budget 时，不裁剪模板或窗口正文，当前窗口模板组失败并提示调整章节参数或模型。
- usage normalize：读取 `prompt_tokens`、`completion_tokens`、`prompt_cache_hit_tokens`、`prompt_tokens_details.cached_tokens`。
- runtime 事件顺序：开始、窗口开始、模板命中、模型调用、usage 更新、工具调用、文件写入、窗口完成、任务完成。
- 工具写入边界：拒绝 `..`、绝对路径、UNC 路径和非本轮报告文件。
- 工具配置：`enabledTools = []` 时启用默认全工具，并写入任务工具快照。
- 工具 schema：导出顺序稳定，`toolSchemaHash` 对同一工具集保持稳定。
- 工具发送时机：路由模型调用不发送工具列表；正式提取模型调用每轮发送同一份冻结后的稳定工具 schema。
- 读工具范围：默认只能读取/搜索当前窗口文本、运行级规则快照、当前书籍 reports 根目录（仅 grep）和本批选中输出报告；拒绝 `runs/<jobId>` 根、未来窗口、trace 日志、项目根、绝对路径、其它 bookId reports 和未选中输出名，拒绝作为 recoverable tool result 返回给模型。
- 工具执行：连续只读工具可并行，写工具和未知工具按模型顺序串行。
- 工具输出预算：`read_file`、`grep`、`glob` 和单次工具结果超限时可截断并记录事件。
- 日志持久化：默认保存结构化事件、hash、文件名、usage 和错误摘要，不长期保存完整 prompt、窗口正文和完整工具结果。
- Prompt budget：达到 warn 阈值记录日志，达到 hard 阈值使当前窗口失败且不静默裁剪窗口正文。
- 账本 key：按 `bookId + windowId + windowHash + templateGroupId + templateGroupHash + outputFileNamesHash + rulesSemanticHash` 判断跳过，不只按窗口编号判断。
- 重叠章节：重叠内容进入 `window-xxxx.txt`；只要窗口文件变化，`windowHash` 变化并触发重跑。
- 文件级账本：模板组内每个输出文件记录 `updated / no-update / skipped / failed`，任一失败则模板组不可跳过。
- `NO_UPDATE` 粒度：零写入时最终文本必须精确为 `NO_UPDATE`，且所有预期输出文件都已查询；有写入时按文件级状态完成，不要求最终文本为 `NO_UPDATE`。
- 部分文件未处理：模板组内预期输出文件未查询时，不能自动记为 `no-update`，必须继续工具循环或失败。
- 重复完整跳过：模型返回 `NO_UPDATE` 时记录 no-update。
- `跳过已提取` 勾选：账本 hash 一致时跳过模型调用。
- `跳过已提取` 取消勾选：即使账本 key 一致也重新进入模型工具循环。
- 不完整补充：模型先 read/grep，再 edit/multi_edit，报告内容更新。
- 未查询直接写入：当前模板组失败，任务进入 `blocked_for_user`。
- 运行错误暂停：除 `RouteFailurePolicy` 可处理的路由错误外，模型协议、工具边界、写入冲突、prompt hard 超限等错误都记录日志并进入 `blocked_for_user`，用户处理后可继续。
- 继续前变更检测：模板、模板组、章节参数、模型 ID、规则文档、工具配置或 prompt budget 变化时，生成 `ResumeChangeSetDto`。
- 新规则继续：用户确认后生成新运行快照，从失败窗口或失败模板组继续；取消则保持 `blocked_for_user` 或 `paused`。

## 实施约束检查

通用架构：

- 提取流程必须进入独立 runtime，不继续堆在 `p0Handlers.ts`。
- Renderer 只消费 DTO 和事件。
- LLM、工具、Markdown writer、报告资产注册职责分离。
- 提取规则文档生成是独立服务，不散落在 UI 或 LLM adapter 中。

设计模式适配：

- Provider registry 解析模型。
- Tool registry 暴露工具 schema。
- Event bus 驱动任务状态和 UI 刷新。
- Repository 保存任务、事件、usage 和报告资产。

配置化覆盖范围：

- 模板、输出文档名、运行窗口参数、工具启用列表、路由规则、提取规则文档、跳过已提取策略、usage mapping 都走配置或任务快照。
- 重试次数、`RouteFailurePolicy`、tool loop 最大轮次/超时、工具默认读取行数/匹配数/结果字节、文件名 sanitizer、文件名冲突策略都走配置或运行时策略快照。
- Prompt budget、工具输出预算、只读工具并行数和工具 profile 也必须走配置或任务快照。
- 不把“丹药分析”“势力分析”等业务数据写死在运行时。
- 不把费用作为 UI 字符串硬编码。

## 决策记录

### ADR-001：规则快照和 hash 拆分

- 日期：2026-06-30
- 状态：已接受
- 决策：每次运行生成不可变规则快照 `runs/<jobId>/rules/提取规则.md`，项目级 `rules/提取规则.md` 只作为 latest copy / 人工入口。事件同时记录 `rulesDocumentHash` 和 `rulesSemanticHash`。
- 理由：完整文档 hash 适合审计，但包含 `jobId` 和生成时间，不能用于账本跳过；语义 hash 排除易变元数据，才能避免误判。
- 影响：进度账本、路由缓存和继续前变更检测必须明确选择使用哪一种 hash。

### ADR-002：账本跳过优先于路由调用

- 日期：2026-06-30
- 状态：已接受
- 决策：`skipAlreadyExtracted = true` 且窗口-模板组账本 key 命中时，跳过该组全部模型调用，包括路由调用；剩余候选组无路由缓存或账本命中时才调用路由模型。
- 理由：账本 key 已包含窗口、模板组、输出文件和规则语义 hash，足以证明该窗口-模板组无需重复消耗模型调用。
- 影响：路由 usage 只在实际发生路由模型调用时记录；账本跳过事件必须能解释为什么没有路由调用。

### ADR-003：路由失败策略配置化

- 日期：2026-06-30
- 状态：已接受
- 决策：路由失败处理使用 `RouteFailurePolicy`，至少包含 `maxRetries`、`fallbackStrategy`、`fallbackSource`、`onFallbackNoMatch`。fallback 规则只能来自任务规则快照或运行策略快照。
- 理由：CLI deterministic pre-filter 不能作为运行时代码硬编码关键词表，否则会与模板/规则配置漂移。
- 影响：运行策略快照必须冻结 fallback 策略；测试需要覆盖 fallback 命中、fallback 空命中和人工阻塞路径。

### ADR-004：状态机错误分类

- 日期：2026-06-30
- 状态：已接受
- 决策：recoverable 错误统一进入 `blocked_for_user`，fatal 才进入任务级 `failed`。工具循环校验失败、模型协议错误、写入边界错误、prompt hard 超限等用户可处理问题都属于 recoverable。
- 理由：这些错误通常可通过修改模板、配置、章节参数或报告文件恢复，进入 `failed` 会破坏继续语义。
- 影响：模板组账本可以记录 `failed` 作为不可跳过单元，但任务状态应保持可继续的 `blocked_for_user`。

### 当前开发切片：阶段 1A 启动前产物生成

阶段 1A 的目标是把当前 P0 同步链路补齐到“模型请求前已有可追溯运行产物”。本切片保持现有 `jobs:start` 同步完成语义不变，仍然是一次模型请求完成后返回；只在发起模型请求前生成或复用运行窗口和规则快照。

本切片只生成以下产物：

- `runs/<jobId>/windows/manifest.json`
- `runs/<jobId>/windows/window-xxxx.txt`
- `runs/<jobId>/rules/提取规则.md`
- `rules/提取规则.md`

本切片明确不实现账本跳过、路由模型、tool loop、多 Markdown 写入、事件订阅和后台异步化。这些能力留给后续切片，避免在 `p0Handlers.ts` 中为了当前需求一次性扩大运行时范围。

进入代码前必须先满足配置化门槛：

- `RouteFailurePolicy`、`ruleSections`、模板组默认生成/退化策略必须进入 config/schema/defaults/invariants 或任务快照构造层。
- 桌面端 handler 禁止写死通用抽取规则、写入规则、跳过规则、fallback 策略和模板组 ID。
- 模板组暂时可以按配置化规则退化为一模板一组，或按输出文件名成组；但分组规则和退化策略本身必须来自配置或任务快照。

`sourceTextPath` 的来源也必须在编码前收敛：

- P0 上传结果或书籍记录需要保存或可取回 `sourceRelativePath`，启动窗口生成优先使用该字段解析原文路径。
- 只有历史数据迁移或兼容场景才允许用受控 helper 推导固定上传路径；该 helper 必须集中封装，并有测试覆盖，不能在 handler 中按目录结构临时推导为唯一方案。

### 当前开发切片：阶段 1B 多窗口 no-tool 执行

阶段 1B 的目标是在阶段 1A 已生成或复用 `runs/<jobId>/windows/manifest.json` 和规则快照的基础上，把当前单次模型请求推进到“按窗口顺序多次请求”。当前已落地实现仍保持 `jobs:start` 同步语义：启动后等待所有窗口完成，再返回最终任务状态；暂不引入后台异步 IPC 事件订阅。

阶段 1B 已落地细节：

- desktop `jobs:start` 通过 `apps/desktop/src/main/windowRunService.ts` 接入 `@novel-extractor/jobs` 的 job runtime，由 job runtime 按 manifest 中记录的顺序逐窗口执行。
- 每个窗口发起一次 OpenAI-compatible no-tool 模型请求；请求 payload 不传 `tools` / `tool_choice`。
- raw/window 报告命名来自 `getDefaultConfig().rawWindowReportDefaults`。当前默认值为 `fileNamePrefix: "raw-window"`、`displayNamePrefix: "原始窗口"`；实际文件名形如 `raw-window0001.md`，展示名形如 `原始窗口0001`。
- `ReportAsset` / `ReportDto` 已增加可选 `reportKind?: "raw-window" | "template-output"`；阶段 1B 注册 raw/window 报告时设置为 `raw-window`。
- handler 层对同一 job 的并发 `jobs:start` 复用 active promise，避免重复启动窗口循环和重复发起模型请求。

本切片范围：

- 基于 `runs/<jobId>/windows/manifest.json`，按 manifest 中记录的顺序逐窗口执行，不能重新临时规划窗口顺序。
- 每个窗口发起一次 OpenAI-compatible 模型请求，且请求不携带 tools。
- 每个窗口完成后累计进度和 usage，最终任务卡或日志摘要可显示 `窗口 N/N`。
- 为每个窗口写入一个临时 raw Markdown 报告，并注册到报告列表；文件名和展示名走 `rawWindowReportDefaults`，不在 handler 中写死。
- 窗口循环、模型调用、usage 累计、raw 报告写入和报告注册由 `windowRunService` 承接，handler 只负责校验、调用和同 job active promise 复用。

本切片非范围：

- 不做大模型模板路由。
- 不做进度账本跳过，也不根据账本跳过路由或窗口。
- 不做 tool loop，不暴露或执行 `read_file`、`grep`、`write_file`、`edit_file`、`multi_edit` 等工具。
- 不做多 Markdown 模板写入；raw/window Markdown 只是临时原始产物，不等同最终模板输出 Markdown。
- 不做后台异步 IPC 事件订阅；事件流和实时刷新留给后续切片。

本切片产物：

- 复用阶段 1A 的 `runs/<jobId>/windows/manifest.json`、`window-xxxx.txt` 和 `runs/<jobId>/rules/提取规则.md`。
- 每个窗口生成一个 raw/window Markdown 报告文件，文件名策略来自 `getDefaultConfig().rawWindowReportDefaults` 当前默认值或后续任务快照。
- 报告列表注册每个 raw/window 报告，并用 `reportKind = "raw-window"` 标记为阶段 1B 临时 raw 产物，避免 UI 或后续流程把它误认为模板最终输出。
- 任务状态、进度和 usage 记录按窗口累计；当前 normalized usage 只进入既有 `TokenUsage` 的 input/output/total 字段。

失败语义：

- 任一窗口模型请求、raw 报告写入或报告注册失败时，停止后续窗口。
- 任务进入 `failed`，失败原因记录窗口序号、窗口文件、失败阶段和底层错误摘要。
- 已经写入并注册的 raw/window 报告保留，不回滚、不删除。
- 未执行窗口不得伪造报告、usage 或 completed 事件。

配置化约束：

- 窗口参数、窗口 manifest 路径、raw 报告命名策略、报告类型标记、模型参数和 usage mapping 来自 job input、config 或任务快照。
- 模板、输出名、规则文档路径、fallback 策略和后续模板输出约束来自配置或任务快照；阶段 1B 即使暂不执行模板路由，也不能在 handler 中写死模板或领域规则。
- 禁止把“凡人修仙传”、CLI 模板、领域关键词、固定 raw 文件名前缀、固定输出目录推导散落写入 `p0Handlers.ts`。

后续需补足：

- 后台异步事件流、模板路由、tool loop、账本跳过和多模板 Markdown 输出仍属于后续切片，不应倒写成阶段 1B 已完成能力。

### 当前开发切片：阶段 1C 最小正式工具循环写入

阶段 1C 的目标是在阶段 1B 已按 manifest 顺序执行窗口的基础上，把每个窗口的 no-tool 原始报告推进为正式工具循环：模型不再只返回 raw/window 诊断文本，而是通过受控文件工具读取、搜索、写入或编辑当前书籍 `reports` 目录中的模板 Markdown，最终生成或更新模板对应的正式 Markdown 报告。

本切片不是最终完整 CLI 等价方案。它只补上“模型可通过工具写入正式模板输出”的最小闭环；真正接近 CLI 输出质量仍需要后续的大模型路由、路由缓存、进度账本、模板组拆批策略、后台事件流和可恢复状态机。

本切片范围：

- 每个窗口仍按 `runs/<jobId>/windows/manifest.json` 中的 manifest 顺序执行，不能重新规划窗口顺序。
- 不做路由；默认把当前 job 选中的模板按配置或任务快照组成一个或多个工具循环批次。
- 同一窗口内必须按 `maxFullTemplatesPerCall` 将模板拆成串行 tool loop 子批次；默认值为 1 时，每个子批次只包含一个模板、独立 messages 和独立工具轮次预算。
- 模型请求必须携带 `read_file`、`grep`、`write_file`、`edit_file`、`multi_edit` 等文件工具 schema；请求体中的工具列表来自配置或任务快照，不在 handler 中临时拼装领域规则。
- 工具执行根限制在当前项目和当前书籍 `reports` 目录；每个子批次的可写 outputFileName 只包含本批模板，任何越界读取、搜索或写入都必须被拒绝。
- 写入或更新正式模板 Markdown 后，注册或更新对应 `ReportAsset`，并设置 `reportKind = "template-output"`。
- 模型必须先 `read_file` 或 `grep` 目标 Markdown，再对既有文件执行 `edit_file` / `multi_edit`；新文件允许直接 `write_file`，同一 tool loop 内已成功写入的同一报告可继续用 `edit_file` / `multi_edit` 修正。
- 如果既有报告被模型再次传给 `write_file`，只有本轮已经成功 `read_file` / `grep` 同一报告，且 `content` 完整包含当前磁盘旧内容时，Runtime 才允许安全重写并注册成功写入。
- 未查询既有报告，或查询后 `write_file.content` 不包含完整旧内容时，Runtime 返回可恢复 tool error，并要求模型改用 `edit_file` / `multi_edit` 或提供包含完整旧内容的新版报告；该错误不应放宽 unsafe path、secret path 或其它 invalid args。
- 写工具正文必须通过公开元数据校验；正文中的资料来源/参考范围只能使用窗口编号、章节范围、章节名或原文范围，不得把 `runs/<jobId>/...`、`assets/books/...`、本机绝对路径、AppData 项目路径或“后续窗口”等内部运行/流程信息写进最终 Markdown。该校验不拦截 `oldText`，保证模型可删除旧报告中的脏路径。
- 写工具正文不得包含 `window-000x` 运行窗口文件名/内部窗口标识，不得包含 `状态：草案` 或 `状态：模板`；这些校验只检查新写入正文，不拦截工具 `path` 或 `oldText`。
- Prompt 必须明确声明模板正文只作为结构、字段和写作规则参考，不是正式报告正文；正式报告不得复制模板标题、`状态：模板`、`前置声明`、`参考范围`、示例或占位案例；正式报告标题必须使用 `outputFileName` 去掉扩展名后的报告名，不得保留或添加“模板”。
- Prompt 必须明确声明模板示例、字段说明、示例事件链和通用体系词不能因为出现在模板中就写进正式报告；只有当前窗口原文或已读取既有报告明确证实时才可写。`长期余波`、`可参考点` 等分析字段不得用模板泛化话术推导未来影响。
- usage 累计包含每一轮模型调用；工具调用本身不产生 LLM usage，也不得伪造 usage。
- 工具循环诊断日志按子批次落到 `runs/<jobId>/tool-loop-traces/window-0001/batch-0001.jsonl`；每行只记录结构化摘要，包括 job/window/batch/round、completion 摘要、tool name/id、参数类型摘要、recoverable/fatal error、tool result summary 和 usage。隐私边界必须固定：正文大字段和完整 `content` / `newText` / `oldText` / `preview` 只写长度或 `<omitted>`，所有字符串先做 secret redaction 后再入日志，不把密钥或完整正文落盘。

本切片非范围：

- 不做大模型模板路由，也不引入路由缓存。
- 不做进度账本跳过，也不根据账本跳过窗口、模板或工具循环批次。
- 不做后台 IPC 事件流或实时日志订阅；`jobs:start` 可以继续沿用当前同步完成语义。
- 不做模板组拆批复杂策略；当前实现只按 `maxFullTemplatesPerCall` 顺序切分模板子批次，不做路由命中后的复杂重组。
- 不做完整 `blocked_for_user` 恢复状态；阶段 1C 失败暂可进入任务级 `failed`，但必须记录这是阶段性差距，后续恢复状态机需要把可处理问题升级为可继续状态。

配置化和架构约束：

- 工具列表、工具预算、最大轮次、工具输出预算、报告文件名、模板正文、模板输出名、prompt 包装语句和文件名冲突策略都应来自配置或任务快照。
- 不在 `p0Handlers.ts` 或 handler 相邻流程中硬编码“当前小说”“当前模板”“领域关键词”“固定输出名”“固定 prompt 包装语句”等领域规则。
- 优先新增或复用独立 service 承接正式 tool loop、工具执行、安全边界校验、usage 累计和 `ReportAsset` 注册；handler 只负责校验输入、装配依赖、调用服务和复用同 job active promise，避免继续膨胀 `p0Handlers`。
- 工具注册和执行应保持清晰模块边界：工具 schema 构造、路径规范化/越界拒绝、模型响应解析、文件变更应用和报告资产注册分别可独立测试。
- 若发现子批次实现开始硬编码模板组或输出规则，应先把对应规则下沉到配置或任务快照，再进入编码。

阶段 1C 与阶段 1B raw 报告的关系：

- 阶段 1B raw/window 报告是临时诊断产物，用于观察窗口输入和 no-tool 模型输出，不等同最终模板 Markdown。
- 阶段 1C 可选择继续保留 raw/window 诊断报告，也可以改为只注册正式模板输出；无论选择哪一种，都必须在测试中固定行为，并在 UI / 报告类型上用 `reportKind = "raw-window"` 与 `reportKind = "template-output"` 明确区分。
- 若同时保留 raw 和正式模板输出，报告列表不得把 raw 报告排在或展示成最终抽取结果；正式模板输出才作为用户主要查看对象。

失败语义：

- 任一窗口的模型协议解析、工具越界、工具执行、正式 Markdown 写入或报告注册失败时，停止后续窗口。
- 任务暂可进入 `failed`，错误摘要需要记录窗口序号、窗口文件、模板批次、失败阶段和底层错误。
- 已经成功写入并注册的正式模板输出保留，不回滚、不删除；未执行窗口不得伪造模板输出、usage 或 completed 状态。
- 由于完整 `blocked_for_user` 恢复状态不在本切片内，所有“本可由用户修改模板/配置恢复”的失败都必须在文档或测试命名中标注为阶段 1C 临时处理。

验证标准：

- mock 模型返回多个 tool calls，能够写入多个正式模板 Markdown，并注册为 `reportKind = "template-output"`。
- 第二轮模型调用能先读取或搜索既有 Markdown，再通过 `edit_file` / `multi_edit` 更新既有文件；若使用 `write_file` 重写既有报告，必须证明新版内容完整保留旧内容。
- 模型请求体包含配置/任务快照提供的 tools schema，且不再是阶段 1B 的 no-tool 请求。
- 工具写入、读取或搜索越出当前项目和当前书籍 `reports` 目录时被拒绝，并停止后续窗口。
- 任一窗口失败后不再执行后续窗口，任务状态进入阶段 1C 临时 `failed`。
- 报告列表出现正式模板输出，而不是只出现 raw/window 诊断报告。
- 若保留 raw/window 诊断报告，测试明确 raw 与正式模板输出的注册顺序、展示类型和 `reportKind` 差异。

### 当前真实运行 harness 命名约束

真实模型 harness 的模板源文件和模板输出文件名必须分离：

- 源模板文件仍来自 `NOVEL_EXTRACTOR_REAL_TEMPLATE_FILES` 或默认四个源文件：`NPC性格与代表事件模板.md`、`事件因果链（长程因果图）模板.md`、`势力设定模板.md`、`材料分析模板.md`。
- 保存到桌面端模板的 `fileName` 使用独立输出文件名，默认从源文件名去掉末尾“模板”：`NPC性格与代表事件.md`、`事件因果链（长程因果图）.md`、`势力设定.md`、`材料分析.md`。
- 定制真实 run 时可通过独立配置覆盖输出文件名，输出文件名数量必须与源模板数量一致。
- 真实 run JSON 的 `reportOutputs` 以桌面端报告 DTO 为准，因此应反映不带“模板”的输出文件名；源模板路径只用于审计输入来源。
